## Architecture

Single-activity Jetpack Compose app (minSdk 24, targetSdk 36). No ViewModel, no Navigation Compose — screen state lives in `MainActivity.kt` via `rememberSaveable`. The data layer syncs races/teams/legend/member-tags via signed `GET` with ETag and uses Room as the single source of truth. NFC reader-mode and GPS-track recording are the two device-integration subsystems.

## Conventions & gotchas

These cross-cutting rules hold project-wide; the file bullets below assume them rather than repeating them.

- **minSdk 24, no desugaring** — use `SimpleDateFormat` (`Locale.US`, UTC where needed), never `java.time`. ISO date/time strings are fixed-width, so compare them **lexicographically** instead of parsing.
- **Pure models** (`ScanSession`, `CheckpointColor`, `ThemeMode`, `TrackProfile`, `TrustedClock`, `ProvisioningModel`, `ChipCheckModel`, `TrackModels`, `GpxExport`, the `*Logic`/mapper helpers, `LegendCrypto`) are **Android-free and JVM-unit-tested** (their `*Test` classes). **Compose UI, real Android adapters** (location engines, `TrackRecordingService`, the `NfcA`/DAO adapters), and trivial wiring are **untested by convention**. Trust boundaries (request signing, `ServerTimeInterceptor`) are the exception — they are tested.
- **Duplicate, don't couple** — small repeated UI rows (e.g. `SwitchTeamRow`/`ChangeTeamRow`, the marks vs track upload-status row) are **copied**, not extracted to a shared file.
- **Room is shipped** (see memory `room-released-with-migrations`) — on-device data must survive upgrades. A schema bump needs a real `Migration` appended to `.addMigrations(...)` **and** a committed `schemas/<n>.json`, or it crashes on upgrade. `exportSchema = true`.
- **Overlay pattern** — full-screen overlays (scan, settings, team-picker, admin, provisioning, check-chip, bind/unbind) are driven by `rememberSaveable` flags rendered after `Scaffold` in one `Box`, dismissed via `BackHandler`. No Navigation Compose. Per-team/race flows use the null-guarded keyed-`remember` pattern: `remember(selectedTeamId) { id?.let { repo.observe(it) } ?: flowOf(emptyList()) }`.
- **Writes outlive overlays** — a write triggered by a closing overlay (select team, bind, persist take) runs on `container.applicationScope`, not composition scope.
- **Repo refresh pattern** — on a `200`, persist rows (`replaceAllForRace`) **then** upsert the ETag as two separate transactions (a crash between them keeps fresh data with a stale ETag and self-heals next sync); the ETag upsert is skipped when the response omits the header.
- **SyncSource routing + persist guard** — the four sync repos take a `source: SyncSource = SyncSource.Cloud` param (default preserves every existing call site); a `Cloud` refresh for a currently-pinned race returns `RefreshResult.Skipped` **both** at method entry (before the network call) **and** again in the `200` branch (an in-flight cloud response must not clobber fresher local rows just-pinned mid-flight). Cloud and LAN ETags are partitioned by origin in `sync_meta`, but the data tables they gate (races/teams/legend/member-tags) are **not** origin-scoped — so every `200` write also deletes the *other* origin's cached ETag for that resource (`SyncMetaDao.deleteEtag`), forcing that origin's next fetch past any 304 that would otherwise skip re-persisting over rows the other origin just wrote. See `data/lease/`, `data/SyncSource.kt`, `data/sync/SyncCoordinator.kt`.
- **Dual-target upload loop** (marks + track) — `Mutex.tryLock`-guarded, drains in `LIMIT` batches, marks only `accepted ∩ batch`, scoped by `(raceId, teamId)`, idempotent by client UUID, flushes to both cloud and LAN targets independently.
- **Secrets** — `BuildConfig` fields from `local.properties` keys (env-var fallback for CI); the build fails with a descriptive error if a required key is missing.

## Cross-file wiring (MainActivity)

- **Overlay stack** — all overlays render after `Scaffold` in one `Box`, each gated by a `rememberSaveable` flag (`showScan`, `showSettings`, `showUpload`, `showAdmin`/`showProvisioning`/`showCheckChip`, `bindSlot`/`unbindSlot`, `teamFlowStep`/`confirmTeamId`). Sibling `BackHandler`s are registered in stacking order (later wins) and each fires only when no deeper overlay is up. Flags reset on `onScanClick` and on team switch (`LaunchedEffect(selectedTeamId)`) — `showUpload` mirrors `showSettings` and is **not** reset on team switch (only reachable via Settings). Theme/economy/admin-session are collected **once** here and threaded down as params (no second `collectAsState`). **`lastRealTeamId` guard** — the app is now portrait-locked (`android:screenOrientation="portrait"` on `MainActivity`, see Config/references), which eliminates orientation-triggered Activity recreation, but other config changes (multi-window resize, font-scale) still recreate it and transiently reset `selectedTeamId`'s `collectAsState()` to `null`. A `rememberSaveable var lastRealTeamId` tracks the last genuine non-null team id; the effect body (overlay resets + the recording-stop check) early-returns unless `selectedTeamId` is non-null **and** differs from `lastRealTeamId`, so a transient-null recreate blip no longer closes open overlays (e.g. the photo-capture camera) or drops a still-relevant team's recording.
- **Scan wiring** — the overlay is fed `scanBindings` (`nfcUid → numberInTeam`) **filtered to the current roster's slots** — a correctness guard: a stale binding for a removed member would otherwise let its chip falsely complete a take (`complete` only compares `present.size` to `expectedCount`). The FAB is gated on a resolved team (else routes to the picker); `onScanTag` refuses (`BadKp`) an empty roster. The opening tap and every later tap share one `ScanInput` path (`Captured` reuses the already-read `(code, uid)`, `Live` reads them) → `legendRepo.unlock` (if a code) → `classifyTag` → persist via `MarkRepository`. **`attachLocation` fires only on the new-take branch** (one-shot `currentLocationProvider.current()`, `null` = no-op). `ScanScreen.process()` plays `scanFeedback.play(feedbackFor(event))` after the `currentOnScanTag` result so every tap (incl. idempotent re-taps) fires exactly one of the three outcomes.
- **Open-on-tap** — there is a **single open source**: the live-idle foreground path publishes `CapturedScan(code, uid, sample)` (no NDEF cold/warm launch — raw `K24` chips can't cold-launch the app). A `LaunchedEffect(captured, teamState)` dispatches it: another overlay up → drop; `Present` → open scan; `None`/`Missing` → open the picker; `Loading` → **return without clearing** so the next state transition re-fires. `ScanScreen` drains `pendingScan` so the opening tap is byte-identical to later taps. Auto-close on window expiry and on completion (a ~3.5 s green «Готово!» hold matching the completion fanfare — see the checkpoint-take celebration below).
- **Checkpoint-take celebration** — the tap that transitions a session from incomplete to complete (which can land on a **Kp** event, when members were buffered before the КП tap) plays `scanFeedback.checkpointComplete()` fanfare in place of the ordinary success beep; `ScanScreen`'s `SUCCESS_HOLD_MS` (3 500 ms, the fanfare length) holds the green «Готово!» screen for that whole duration before auto-closing, and only that auto-close path fires `onCompleted` (never expiry or manual close). `MainActivity` sets a one-shot `pendingCelebration` (plain `remember`, not `rememberSaveable` — a recreate mid-celebration deliberately drops it; also cleared on team switch) and navigates to «Отметки» (`switchToTab(0)`). `MarksScreen` consumes `celebration` on arrival: scrolls its `LazyColumn` to the bottom, then plays `onCoinSound()` together with a bouncy scale/fade pop-in on the new last tile, and calls `onCelebrationDone()`.
- **Local-mode switch + PTR re-route** — the Settings «Данные» switch position is **derived, not stored**: `isPinned(container.raceLease.collectAsState(), selectedRaceId ?: nearestRaceId, trustedClock.trusted() ?: wallClock)`, so a handback or expiry flips it live even while Settings is open. `onLocalModeChange` launches `syncCoordinator.enterLocalMode()`/`exitLocalMode()` on `applicationScope` and maps the returned `LocalModeOutcome` to a RU Toast. «Команда»'s pull-to-refresh now calls `syncCoordinator::refreshAll` (intentionally broadens the pull from teams-only to races+teams+legend+member-tags); «Легенда»'s pull passes `source = syncCoordinator.sourceFor(raceId)` (else a pinned pull would silently `Skipped` against the cloud guard); the races picker's pull stays Cloud-only (races isn't race-scoped); `onRaceSelected`'s prefetch routes teams+legend through `sourceFor(raceId)`.

## UI layer

- `MainActivity.kt` — entry point; `HorizontalPager` swipe-tab navigation. **Hosts all overlay state** (see above) and the **NFC reader-mode infra** (`NfcAdapter.ReaderCallback`, enabled in `onResume`/disabled in `onPause`). A set of `@Volatile` tag hooks (`onTagScanned`, `onTagForChipInfo`, `onTagForMark`, `onTagForProvision`, `onTagForVerify`) are armed/cleared by each overlay's `DisposableEffect`; `onTagDiscovered` dispatches them by priority on the binder thread (hence `@Volatile`), and when none is armed the idle path reads `(code, uid)` once and publishes a `CapturedScan` (an unknown idle tag — not a КП, not a bound bracelet — instead plays `scanFeedback.neutral()` before dropping). `READER_FLAGS` carries `FLAG_READER_NO_PLATFORM_SOUNDS` to **suppress the stock Android NFC beep app-wide** — the `ScanFeedbackPlayer` outcomes replace it everywhere. **Deliberate silent paths** (not oversights): a provisioning tap during a pager swipe / busy job, and a recognized chip with no team selected (the picker navigation is the feedback). `nfcState: NfcState { NoHardware, Disabled, Available }` recomputed each `onResume`. `SelectedTeamState { None, Loading, Missing, Present }` drives the «Команда» tab (`Loading` suppresses the empty-state flash). `onRaceSelected` fires an on-tap prefetch of teams+legend on `applicationScope` so the network starts during the screen transition. A `LaunchedEffect` ticks `trustedClock.recomputeStatus()` every 5 s. **5-minute foreground upload retry:** next to that ticker, a `LaunchedEffect(Unit)` wraps `lifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) { while (true) { … ; delay(UPLOAD_RETRY_INTERVAL_MS) } }` (`UPLOAD_RETRY_INTERVAL_MS = 300_000L`) — each tick launches `trackRepo.uploadAllPending()` and `markRepo.uploadAllPending()` on `container.applicationScope` (fire-and-forget, survives backgrounding mid-batch). `repeatOnLifecycle` cancels the block when the Activity leaves `STARTED` and restarts it on return to foreground, so attempt-before-delay means a return-to-foreground is an immediate retry (deliberate); overlap with Launch B / the track service's 10-min loop / take-complete flush is safe via the repos' `Mutex.tryLock`. **`uploadRefreshing` PTR host wiring:** a `remember`ed (not `rememberSaveable` — transient) `var uploadRefreshing` near `showUpload` backs `UploadScreen`'s `refreshing`/`onRefresh` params — `onRefresh` guards on `!uploadRefreshing`, runs both `uploadAllPending()` calls in a `supervisorScope` (mirrors the Launch B precedent — one target failing doesn't cancel the other) on `applicationScope`, and clears the flag in `finally` (so a thrown repo call still stops the spinner).

Pure `internal tileFill(color, darkTheme)`/`gridGrout(darkTheme)` carry a **muted, screen-scoped** palette distinct from the bright `CpColor*` bars; both take a plain `darkTheme: Boolean` and the caller resolves the **applied** theme via `MaterialTheme.colorScheme.surface.luminance() < 0.5f`, **not** `isSystemInDarkTheme()` (or a manual theme override would mismatch). `null`/unknown color → the only theme-flipping neutral fill (so a colorless tile never glows). **«ВЗЯТО» counts only scoring КП** (`cost > 0`): `takenPointCount(marks, costOf)` filters distinct `complete` takes on `costOf(it) > 0`, denominator is the server-derived `legendScoringCount` (see `LegendRepository.scoringCountForRace`) — technical CPs (`cost = 0`, e.g. test/transfer zones) never inflate the counter. - `ui/scan/ScanSession.kt` — pure scan-session model; defines `SCAN_WINDOW_MS = 20_000L`. `reduce(session, event, now)` is the only state machine: `Kp` drains the pre-KP buffer into `present` and re-stamps; a repeat `Kp` preserves `present`; a **KP switch** resets `present` (so the prior КП's members aren't credited to the new one); `Member` buffers while no КП else joins `present` (set-semantics). `classifyTag(...)` maps a non-null code → `Kp` (via the `unlock` outcome) or a bracelet uid → `Member`/`UnboundChip`. `isComplete(...)` is a **cosmetic dismiss gate only** — scoring is persisted independently. Tested by `ScanSessionTest`/`ScanTagDecisionTest`.
- `ui/scan/ScanFeedback.kt` — pure scan-feedback mapper. `enum ScanFeedbackKind { Success, Failure, Neutral }` + `feedbackFor(event: ScanEvent): ScanFeedbackKind` — **exhaustive over `ScanEvent` (no `else`)**: `Kp`/`Member` → `Success`, `UnboundChip`/`BadKp` → `Failure`. **`Neutral` is never returned by the mapper** — it is a non-overlay outcome produced only by direct `neutral()` calls at the idle/bind/check branch points ("duplicate, don't couple"), so the mapper stays a clean 2-way over `ScanEvent`. Tested by `ScanFeedbackTest`. The outcomes replace the suppressed platform NFC beep. **Sound is reserved for working chips** (`Success`/`Failure` play a clip); `Neutral` (unknown chip) is **vibration-only, no sound**. Completion of a full КП take is **not** modeled in this mapper — it's a session-transition check in `ScanScreen.process()` that calls `scanFeedback.checkpointComplete()` directly (see the checkpoint-take celebration bullet above), keeping this mapper a clean 2-way over `ScanEvent`.

- `data/ScanFeedbackPlayer.kt` — Android adapter playing the scan outcomes plus two celebration cues. Loads five clips in `init`: `beep_ok3`/`beep_err` (`success`/`failure`), `shutter` (photo-capture click), `checkpoint_mark_completed` (`checkpointComplete()` fanfare), `mark_added_mario` (`coin()` tile pop-in). **Sound is for working chips only:** `success()`/`failure()` play a clip + vibrate `SUCCESS_VIBRATION_PATTERN`/`0,60,80,60`; `neutral()` (unknown chip) is **vibration-only — no sound**; `shutter()` is sound-only (no vibration, photo-mark only); `photoCaptureConfirm()` is vibration-only (`SUCCESS_VIBRATION_PATTERN`, the audible cue already fired at `shutter()`); `checkpointComplete()` plays the fanfare **and** the same `SUCCESS_VIBRATION_PATTERN` (it replaces, not adds to, the ordinary success beep on the completing tap); `coin()` is sound-only (a UI animation cue, not a chip confirmation). `play(kind: ScanFeedbackKind)` dispatches `feedbackFor`'s three outcomes only — `checkpointComplete()`/`coin()`/`shutter()`/`photoCaptureConfirm()` are called directly by their call sites, not via `ScanFeedbackKind`. Owned by `AppContainer` and constructed **eagerly** in `Application.onCreate` so the asynchronous `SoundPool.load()` of all five clips (`USAGE_ASSISTANCE_SONIFICATION`, `maxStreams = 2` — fanfare and coin are sequenced by the celebration flow, never concurrent with each other) finishes before the first NFC tap; `SoundPool.play()` on an unloaded id returns 0 (silent no-op), so a missed sound on a very early tap is acceptable. **Vibration always fires regardless** — a missed vibration is not acceptable. `Vibrator` resolved via `VibratorManager` (API 31+) / `VIBRATOR_SERVICE` (below), each `vibrate(...)` split `VibrationEffect.createWaveform` (API 26+) / deprecated `vibrate(pattern, -1)` — every branch guarded by `Build.VERSION.SDK_INT` for `NewApi` lint. **`android.permission.VIBRATE` must stay in the manifest** or every vibrate is a silent no-op. `SoundPool.play`/`Vibrator.vibrate` are thread-safe → callable from the NFC binder thread and coroutines. **Untested by convention** (Android adapter).
- `data/track/` — GPS track subsystem. `TrackModels.kt` (pure `RawFix`, `TrackPointLike`, `filterPoints` >50m accuracy drop for **read only**, reboot-safe `sortedTrackPoints`, upload-status value types + pure `uploadResultKind`); `GpxExport.kt` (pure GPX 1.1, **one `<trkseg>` per `segmentId`** so a stop→start gap isn't a teleport line); `TrackState.kt`; `TrackRepository.kt` (single source of truth; owns the impure `RawFix → TrackPointEntity` mapping — back-projects `wallMs`/`trustedMs` to the **fix moment**, not the insert time; dual batch upload independent per target; `uploadLoop` returns `UploadResultKind?` where an idle re-flush returns `null` so it can't overwrite a real error with a misleading "ok", and a `Success` with no forward progress hand-maps to `Error`); `TrackProfile.kt` (pure `Precise` 15 s / `Economy` 180 s) + `TrackProfilePreference.kt`; `LocationEngine.kt` (seam; pure `chooseEngineType`, untested Fused/Legacy adapters; `flush(onComplete)` forces buffered delivery before stop — fixes a «Стоп»-drops-last-batch bug); `CurrentLocationProvider.kt` + Fused/Legacy adapters (one-shot fresh fix for the take coordinate — **never throws** (any failure → `null`; "no coordinate" beats a stale one), freshness mandatory, drops fixes older than `MAX_FIX_AGE_MS`). Pure parts tested (`TrackPointMappingTest`/`UploadOutcomeTest`/`GpxExportTest`/`TrackProfileTest`/`LocationEngineFactoryTest`/`TrackRepositoryTest`).
- `data/nfc/MifareUltralightWriter.kt` — chip provisioning over a raw header-first on-chip format (no NDEF): page 4 = `K24` magic + packed version/type, pages 5–8 = the 16-byte code (`bid = sha256(code)[:16]`). **Gotchas:** writes the header **last** (commit-marker order) so an interrupted tap can't leave a valid header over a half-written code; reads back over the **same** open transport (only one `TagTechnology` connection allowed at a time); uses `NfcA` directly, **not** `MifareUltralight.get()`, because Android omits that tech for many NTAG/Ultralight chips; read tries FAST_READ then falls back to plain READs, never throwing. The `NfcTransport` fun-interface seam makes sequencing JVM-testable (`MifareUltralightWriterTest`); the thin `NfcA` adapter is not tested.
- `data/crypto/LegendCrypto.kt` — pure, offline, Android-free crypto for the encrypted legend (mirrors the server; spec in `docs/design/API.md`). `bid`, hand-rolled HKDF-SHA256, AES-256-GCM `open`, and `unlock(...)` doing the `bundle_blob → content_key → enc` indirection — **never throws** (catches → `Failed`). **Base64 is okio `ByteString`** (not `java.util.Base64` (API 26) or `android.util.Base64` (JVM-stubbed)). `LegendCryptoSanityTest` covers round-trips; `LegendCryptoTest` (server-vector assertions) is `@Ignore`d until a server fixture lands — the engine ships **crypto-unverified against the server** until then.
- `data/time/TrustedClock.kt` — pure trusted-time core. Derives **trusted** time from the monotonic `elapsedRealtime()` + server time (`trusted = serverEpochMs + (elapsedNow − anchorElapsedMs)`) — immune to wall-clock changes; only a **reboot** breaks it (detected by monotonic regression and/or `BOOT_COUNT` mismatch). `TimeSample` is one atomic snapshot of all time sources; `signingSeconds()` feeds request `ts`; `trustedAt(elapsedAt, bootAt)` extends the formula to an arbitrary past monotonic moment (the per-fix GPS use). **Warm start through boot identity** (`null == null` does NOT grant warm start — would falsely verify a reboot). Lock-free reads via `AtomicReference`; all writes under one lock; `persist` is best-effort (never throws on the OkHttp thread). Tested by `TrustedClockTest`.
- `data/AdminAuthRepository.kt` + `data/AdminTokenStore.kt` — race-admin session. `AdminSession { LoggedOut | LoggedIn }` seeded **synchronously** from the store (a past-expiry stored session → `LoggedOut`+cleared); `token()` is a synchronous `.value` read (so the OkHttp thread can call it). `login`/`logout` (logout always clears locally even offline)/`onUnauthorized`. Pure `isExpired` is a **lexicographic** UTC-string compare. The 30-day bearer is revocable, so plain prefs is acceptable. Tested by `AdminAuthRepositoryTest`/`AdminTokenStoreTest`.
- `data/ThemePreference.kt` / `data/TrackProfilePreference.kt` / `data/time/ClockAnchorStore.kt` / `data/InstallId.kt` — pure injected `load`/`save` stores + thin `fromSharedPreferences` adapters, read **synchronously** at construction (no cold-start flash). `ClockAnchorStore` writes the anchor as **one delimited key** (atomic — a kill mid-write can't leave a half-written mix). All pure parts unit-tested.
- `data/NfcUid.kt` — pure `normalizeNfcUid(raw: ByteArray): String` (per-byte zero-padded hex, uppercased, matching the server pool format). Tested by `NfcUidTest`.
- `data/DateUtils.kt` — `todayIso()` + `nearestRaceId(races, today)` (soonest current race), both `SimpleDateFormat`-based.
- `TrackRecordingService.kt` — foreground service (`type=location`, `START_NOT_STICKY` — a kill does not silently resume). Re-checks location permission first thing (TOCTOU on a revoke). Reads the active `TrackProfile`, mints one `segmentId` per recording session (pure `nextSegmentId`), batches `RawFix → trackRepository.insertAll`, and **flushes the buffer before stop** (`flushThen` runs `after` exactly once via callback or a ~4 s timeout fallback — fixes a «Стоп»-drops-last-batch bug). **Live-apply** profile toggle via a `profileJob` soft engine restart (no track gap). **Throttled live upload** piggybacks `trackRepository.uploadPending` + `markRepository.uploadPending` every `LIVE_UPLOAD_MIN_INTERVAL_MS` (10 min) via pure `shouldLiveUpload` (nullable "never uploaded" sentinel so the first batch always fires). **Guarantee:** «Стоп» + live-switch are lossless; a bare `onDestroy()`/hard kill may lose up to `maxDelay` (best-effort). Pure helpers tested (`SegmentIdTest`/`LiveUploadThrottleTest`); the service is not.
