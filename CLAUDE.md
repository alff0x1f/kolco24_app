## Architecture

Single-activity Jetpack Compose app (minSdk 24, targetSdk 36). No ViewModel, no Navigation Compose — screen state lives in `MainActivity.kt` via `rememberSaveable`.

**Current structure:**
- `MainActivity.kt` — entry point; uses `HorizontalPager` (`rememberPagerState`) for swipe tab navigation; `NavigationBar` items call `pagerState.animateScrollToPage()`; `showScan: Boolean` via `rememberSaveable` drives the scan overlay; **NFC reader-mode** infra lives here (`NfcAdapter.ReaderCallback`): a lazy `NfcAdapter`, `enableReaderMode(this, this, READER_FLAGS, null)` in `onResume` / `disableReaderMode` in `onPause`, `READER_FLAGS = FLAG_READER_NFC_A|B|F|V or FLAG_READER_SKIP_NDEF_CHECK`; `AndroidManifest.xml` declares `<uses-permission android:name="android.permission.NFC" />` and `<uses-feature android:name="android.hardware.nfc" android:required="false" />` — `required="false"` keeps the app installable on NFC-less devices, with `NfcState.NoHardware` gracefully disabling bind affordances; `@Volatile var onTagScanned: ((String) -> Unit)? = null` is a nullable hook invoked on the main thread (`mainHandler.post`) with `normalizeNfcUid(tag.id)`; `@Volatile var onTagForChipInfo: ((Tag) -> Unit)? = null` is the **highest-priority** hook for the debug «Инфо о чипе» (GET_VERSION) flow; `@Volatile var onTagForMark: ((Tag) -> Unit)? = null` is the «Отметить КП» scan hook (also raw `Tag`, since marking needs `readChipCode` + `unlock` off the uid); `@Volatile var onTagForProvision: ((Tag) -> Unit)? = null` is the admin bulk chip-provisioning hook (raw `Tag`, since binding needs to bind + write onto the tag), armed by `ProvisioningScreen`; `@Volatile var onTagForVerify: ((Tag) -> Unit)? = null` is the admin read-only chip-verification hook (raw `Tag`, since verification needs `readChipCode` + `LegendCrypto.bid` off the tag — no write, no bind), armed by `CheckChipScreen`. `onTagDiscovered`'s priority is **`onTagForChipInfo` → `onTagForProvision` → `onTagForVerify` → `onTagForMark` → `onTagScanned` → idle**: each posts to the main thread and returns; if none is set, the idle foreground path recognizes our own chips and opens «Отметить КП»: it reads `(code, uid)` once (`readChipCode(tag)` over raw NfcA + `normalizeNfcUid(tag.id)`), and if `code != null` (a КП chip) **or** `uid in boundUidsSnapshot` (a bound team-member bracelet) publishes a `CapturedScan(code, uid, trustedClock.sample())` to `nfcLaunchScan` (the `TimeSample` is snapshotted on the binder thread at tap time) — else drops the tag silently (see «Chip-scan opens «Отметить КП»» below). The `(code, uid)` read here is reused downstream — no second chip read. `onTagForProvision`/`onTagForVerify` never co-arm (the admin home opens one sub-overlay at a time), so their relative order is cosmetic. All hooks are `@Volatile` because `onTagDiscovered` reads them from the binder thread while `DisposableEffect` writes them from the main thread. Provisioning (under `showAdmin`/`showProvisioning`) and the debug chip-info flow (under `showSettings`) are never armed simultaneously, but the distinct hook keeps each `DisposableEffect` owning exactly one hook. `AndroidManifest.xml` declares `android:launchMode="singleTop"` on `MainActivity` — **kept harmlessly** (it originally backed `onNewIntent` NDEF re-delivery, now gone with the raw-chip migration). **The NDEF cold/warm-launch path is removed** (raw `K24` chips carry no NDEF): there is no `NDEF_DISCOVERED` intent filter, no `handleNfcIntent`, and no `CapturedScan(openOnly = true)` publish anymore — the «Отметить КП» overlay opens only via the live reader-mode path (`onTagDiscovered` while the app is foreground) or the orange FAB. (Trade-off accepted: tapping a КП chip with the app fully closed no longer auto-launches it.) `nfcState: NfcState { NoHardware, Disabled, Available }` (top-level enum) is recomputed each `onResume` (`adapter == null`→NoHardware, `!isEnabled`→Disabled, else Available) and read by composables. The bind-chip flow (see «Bind-chip overlay» below) registers/clears `onTagScanned` via `DisposableEffect`; the debug «Инфо о чипе» flow registers/clears `onTagForChipInfo` via its own `DisposableEffect` (DEBUG-only — armed via `onReadChipInfo`, reads `readChipVersion(tag)` off-main and surfaces `chipModelFromVersion(...)`). `SelectedTeamState { None, Loading, Missing, Present(team) }` private sealed interface drives the «Команда» tab: `produceState` switches from `Loading` (initial) to `None`/`Present`/`Missing` as `observeTeam` emits — `Loading` suppresses the empty-state flash before Room delivers the first value; `todayIso()` (the shared top-level helper in `data/DateUtils.kt`, `SimpleDateFormat` not `java.time.LocalDate` for minSdk-24/no-desugaring compatibility) is passed down to `CompPickerScreen`. `onRaceSelected` fires an **on-tap prefetch** before opening the team picker: `container.applicationScope.launch { teamRepo.refreshTeams(raceId) }` + `legendRepo.refreshLegend(raceId)` (applicationScope, NOT composition scope, so the warm-up outlives the closing comp picker) — the network starts during the screen transition. `TeamPickerScreen.onRefresh = teamRepo::refreshTeams` stays the authoritative UI-state driver; the rare duplicate GET with the on-tap prefetch is accepted (idempotent/ETag-guarded)
- `ui/marks/MarksScreen.kt` — Отметки tab on **real take data**: stateless `MarksScreen(marks: List<MarkEntity>, checkpointColors: Map<Int, String>, nfcAvailable: Boolean, onScanClick, modifier)` (no more `MOCK_MARKS`). Orange FAB triggers ScanScreen overlay via `onScanClick`. Pure top-level `marksToTiles(marks, colorOf): List<Mark>` maps **one tile per event** (a repeat take of the same КП is a separate tile), expects newest-first as `observeMarks` delivers and returns them **oldest-first** (`asReversed`) so each new take appends to the end of the grid; `time = SimpleDateFormat("HH:mm", Locale.US)` formatted from `trustedTakenAt ?: takenAt` (the trusted take time when synced, wall fallback otherwise — matching the `COALESCE` order in `MarkDao.observeForTeam`; no `java.time`), `number = checkpointNumber.padStart(2,'0')`, `kind = PHOTO` iff `method=="photo"` else `NFC`, `color = colorOf(mark)` (default `{ null }` keeps the pure mapping testable without a checkpoint map). The screen resolves `colorOf` from `checkpointColors[mark.point]` via `parseCheckpointColor` — the КП color token joined off the mark's `point` in `MainActivity` (race-scoped public data, no migration). **Tile design ("scorecard"):** each `ScorecardTile` is a `surfaceContainerLowest` card (`aspectRatio(1f)`, 10dp radius, hairline `outlineVariant` border) with a **top color stripe** (6dp, `CheckpointColor.barColor()` — a third private copy of the mapping in `LegendScreen.kt`/`ProvisioningScreen.kt`; `null`/`""` → transparent, neutral tile) echoing the physical КП's reflective stripe, overlaid (not a sibling) so it never shifts the centered number. `NfcTileBody` = mono number centered on the **whole** tile (`includeFontPadding=false` + trimmed/centered `lineHeightStyle` so it sits optically centered, not pushed down by font padding), `+cost` top-end, `time` bottom-center — all per-element padding so the stripe/number don't fight. `PhotoTileBody` keeps the same frame: charcoal photo-seat gradient + `MiniCpBadge` (white КП badge w/ red stripe) + `+cost` + `time` on the scrim. The old recessed-reader-plate skeuomorphism (hatching, bevels, wave glyph) and the `isRecent` newest-tile highlight are **gone** — variation now comes from the data (numbers/costs/colors/times), not texture. Metrics: «ВЗЯТО» = `takenPointCount(marks)` (distinct `complete` points), «СУММА» = `totalScore(marks)` (Σ cost over distinct scored points — a repeat take does not double-count; both live in `MarkRepository.kt`), «ДО КВ» is a `"—"` placeholder (no control-time source yet). **Empty state** (`MarksEmpty`, shown when no tiles): a state-aware "next step" guide rather than flat text — takes `hasTeam`/`memberCount`/`boundCount`/`nfcAvailable`/`nfcDisabled` + `onChooseTeam`/`onBindChips`/`onOpenNfcSettings` and picks one of five messages **in workflow order**: no team → «Выбрать команду» (CTA → `onChooseTeam`, comp picker); **NFC switched off** (`nfcDisabled`, checked **before** `!nfcAvailable` so it wins) → «NFC выключен» + CTA «Включить NFC» → `onOpenNfcSettings` (`Settings.ACTION_NFC_SETTINGS` deep-link, same as `BindChipSheet`); NFC absent / no-hardware (`!nfcAvailable && !nfcDisabled`) → photo-fallback note (**folds in** what was a separate floating `NfcUnavailableBanner` — the banner now renders only when tiles exist, and is itself tappable-into-settings when `nfcDisabled`, static «NFC недоступен» when no-hardware); chips not all bound (`boundCount < memberCount`) → «Привязать чипы участникам» + «с чипом N из M» + CTA → `onBindChips` (`pagerState.animateScrollToPage(2)` to the Команда tab) — surfaced because a take only `complete`s once **all** members are present, so an unbound roster never produces a tile; else ready → «Приложите телефон к метке КП». The signature is `ScorecardGhostRow` — a row of 4 ghost `GhostTile`s previewing the grid (first solid w/ the state's lead glyph + neutral top stripe, echoing `ScorecardTile`; trailing three hairline **dashed** placeholders fading out via `drawBehind`+`PathEffect`). `MainActivity` passes `boundCount = teamForTab.members.count { bindings.containsKey(it.numberInTeam) }` (same source as `TeamHeroCard`'s «N / total с чипом»). `marksToTiles` (incl. `colorOf` and the `trustedTakenAt ?: takenAt` time pick)/metrics covered by `MarksMappingTest`
- `ui/legend/LegendScreen.kt` — Легенда tab; stateless: `LegendScreen(checkpoints: List<CheckpointEntity>, hasTeam, onChooseTeam, modifier)` renders two states — `!hasTeam` → **LegendNoTeam** (map glyph + «Выбрать команду» → `onChooseTeam`, same callback the Команда/Отметки empty states use); else the CP list (**LegendList**) fed from Room. The old `LegendLocked` "ДО СТАРТА" card and the `legendVisible` param are **gone** — the legend is now always returned and locked checkpoints are masked per-row instead (the server-level `is_legend_visible` flag was removed). `CheckpointRow` branches on `cp.locked` → **masked** (Lock glyph in the label slot, mono `?-${number.padStart(2,'0')}`, subtitle «Откроется на КП», muted `onSurfaceVariant`, no check icon); else the open/revealed row. `ScoreCard` sums known costs only (`mapNotNull { it.cost }`), and a `lockedCount = checkpoints.count { it.locked }` drives a quiet «+N закрытых КП» hint under the progress bar (shown only when `> 0`). A row is taken iff `cp.id in takenIds`; switching teams within a race shows each team's own progress. **Leading color bar:** `CheckpointRow` wraps both `OpenCheckpointRow` and `LockedCheckpointRow` in a `Modifier.height(IntrinsicSize.Min)` row with a fixed 4dp leading `Box` gutter — `parseCheckpointColor(cp.color)?.barColor()` (the `CheckpointColor.barColor(): Color` Compose mapping lives **here**, not in the pure parser file; uses fixed `Tertiary`/`OrangeCta`/`CpColor*` palette constants — same shade in light & dark, no dynamic theme token passed) tints it, `null` (`""`/unknown) → `Color.Transparent` so neutral rows look exactly as before (the gutter is always present, inner rows lost 4dp of leading padding and the divider start is 72dp so absolute text alignment is unchanged). - `ui/scan/ScanScreen.kt` — **live stateful** full-screen overlay for NFC checkpoint marking (A3 screen). `ScanScreen(roster: List<TeamMemberItem>, chipNumbers: Map<Int, Int>, nfcAvailable: Boolean, onScanTag: suspend (ScanInput, TimeSample) -> ScanEvent, onClose, modifier)` (no more `MOCK_SCAN_CHIPS`). The old `Long` tap-time was widened to a `TimeSample` (trusted/wall/elapsed/boot snapshot from `TrustedClock`) so the take can persist trusted time and the window can run on the monotonic `elapsedMs`. `chipNumbers` maps `numberInTeam → participantNumber` (the chip's pool number) for display — a scanned member's subtitle shows «№{participantNumber}», never the raw `nfcUid`; `MainActivity` derives it (`scanChipNumbers`) with the same team/roster filter as `scanBindings`, which stays uid→slot for `classifyTag`. The diagnostic-surfacing + expired-session-discard + `reduce` body is factored into a local `suspend fun process(input: ScanInput, sample: TimeSample)` (runs inside a `Mutex` that serializes overlapping binder-thread taps; `now = sample.elapsedMs` is the monotonic «сейчас»): `onScanTag(input, sample)` → `UnboundChip`/`BadKp` set a `diagnostic` string instead of advancing the window, else clear `diagnostic`, discard a UI session whose 20 s window already elapsed at tap time, and `session = reduce(...)`. A `DisposableEffect` arms/clears `activity.onTagForMark` (each live tap snapshots `activity.trustedClock.sample()` on the main thread → `process(ScanInput.Live(tag), sample)`); on arm it also **drains** `activity.pendingScan` once (the tap that opened the overlay) as `process(ScanInput.Captured(code, uid), scan.sample)`, then nulls `pendingScan` — so the opening tap is byte-for-byte identical to subsequent taps. A `LaunchedEffect(session?.lastScanAt)` ticks every 250 ms computing `remaining = SCAN_WINDOW_MS(20s) − (SystemClock.elapsedRealtime() − lastScanAt)` (the **monotonic** timer, the same source as `sample.elapsedMs`, so the ring can't jump on a clock change); `<= 0` → finalize inside the mutex (tracking whether it actually fired so a window-extending scan is not closed) then **`currentOnClose()` after releasing the mutex** (`currentOnClose` is `rememberUpdatedState(onClose)`) — expiry now auto-closes the overlay. **Auto-close on completion:** `LaunchedEffect(allScanned)` where `allScanned = isComplete(session, roster.size)` — guarded `if (allScanned && !completed)`, sets `completed = true` before `delay(SUCCESS_HOLD_MS = 1_000L)`, swaps the hero card to a green «Готово!» check state, then `finalizeSession()` + `currentOnClose()`; `completed` resets in `finalizeSession()`. A FAB-opened overlay with a `null` session has no running timer and `isComplete` false → stays open until manual close (unchanged). «Готово» is enabled while `session?.point != null` (a КП must be identified before the button activates) and finalizes + `onClose`; a window expiring with `point == null` commits nothing. `SCAN_WINDOW_MS` lives in `ScanSession.kt` (shared with `MainActivity`). Renders `CpWaitingCard` (point null → «приложите чип КП», else number+cost), a `ChipGrid` built from `roster` (green check = present, «Чип не привязан» for unbound), a `HeroTimerCard` with the live ring/`remaining`, and a `ScanClockBanner` (from `ui/common/ClockWarningBanner.kt`, fed `activity.trustedClock.status`) — a bright `Skewed` accent **and** the soft `NoSync` notice (the only place `NoSync` is surfaced; the take is written either way). Pure session logic lives in `ui/scan/ScanSession.kt` and is unit-tested (`ScanSessionTest`/`ScanTagDecisionTest`); the host UI is untested
- `ui/common/ClockWarningBanner.kt` — clock-skew UI for `TrustedClock`. `ClockWarningBanner(status: ClockStatus, modifier)` is the **global** nag rendered under the per-tab `TopAppBar` (in a `Column` over the `HorizontalPager`, with `statusBarsPadding`): shows **only** on `Skewed` (errorContainer plate «Часы телефона расходятся с сервером на N мин — проверьте дату и время»); `NoSync`/`Ok` render nothing globally (single-visibility rule P2 — the soft `NoSync` notice lives only in the scan overlay, where time is written). `ScanClockBanner(status, modifier)` is the scan-overlay variant: bright `Skewed` accent + a **soft** `NoSync` plate. Pure `formatSkewMinutes(skewMs): String` = «N мин» via `Math.round(abs(skewMs.toDouble()) / 60_000.0)` (round not ceil; `abs` in `Double` so `Long.MIN_VALUE` can't trap; direction-less). `MainActivity` collects `container.trustedClock.status` and drives a `LaunchedEffect { while(true){ delay(5000); trustedClock.recomputeStatus() } }` local tick so skew surfaces within ~5 s of a wall-clock change without any network event. - `data/api/AppSignatureInterceptor.kt` — OkHttp interceptor; HMAC-SHA256 signs every request (canonical string from `docs/API.md`), adds the six `X-App-*`/`X-Install-Id` headers; `ts`/signature recomputed per call (survives retries). - `data/api/ServerTimeInterceptor.kt` — OkHttp interceptor that re-anchors `TrustedClock` from the HTTP `Date` response header on every **network** response (free, backend-free server-time source — no protocol change). Snapshots `elapsed()` on both sides of `proceed()`; `rtt = elapsedAfter − elapsedBefore`; pins the parsed `Date` epoch to an **overflow-safe RTT-corrected midpoint** `anchorElapsed = elapsedBefore + rtt / 2` (not `(before+after)/2`, which could overflow `Long`). Re-anchoring is gated so only a genuinely live `Date` is trusted: **network only** (`response.networkResponse != null` — a cache hit's `Date` is stale), **RTT in `0..maxRttMs`** (default 10 s; a **negative** RTT is a clock/timing anomaly and an over-long RTT makes the midpoint too coarse — both dropped), and **`Date` present+parseable** (`headers.getDate("Date")` → `null` for missing/malformed leaves the anchor untouched). Trust is **structural**: installed only on the single-host `/app/` client (`BuildConfig.API_BASE_URL`/HTTPS), so no `request.url.host` gate — KDoc records that a second host on this client would require adding one. Read-time `filterPoints` (`TrackModels.kt`) still trims coarse fixes by **accuracy** (>50m) only — reversible, raw kept. `LocationEngineFactory` splits the pure `chooseEngineType(gmsAvailable): EngineType` (Fused iff GMS, JVM-tested by `LocationEngineFactoryTest`) from the thin `create(context, profile: TrackProfile)` adapter (`GoogleApiAvailability.isGooglePlayServicesAvailable == SUCCESS`, passes `profile` into the chosen engine, untested)
- `data/NfcUid.kt` — pure `normalizeNfcUid(raw: ByteArray): String`: hex-encode each byte (zero-padded, `0x04`→`"04"`) then `.uppercase()`, matching the server pool format. No `String` overload (pool values arrive already normalized). Tested by `NfcUidTest`.
- `data/nfc/MifareUltralightWriter.kt` — chip provisioning primitives. **Raw header-first on-chip format** (no NDEF): page 4 = 24-bit magic `K24` (`0x4B 0x32 0x34`, the Kolco24 brand "this is our chip" sentinel) + a packed `version<<4 | type` byte (`0x11` = version 1, type КП); pages 5–8 = the 16-byte code (`bid = sha256(code)[:16]`, unchanged). `ChipWriteResult { Success / Unsupported / Failed(message) }` sealed interface; never throws. **Pure helpers** (JVM-tested, no Android): `buildChipRecord(type: Int, code): ByteArray` (`require` 16-byte code + nibble `type`; magic + packed byte + code = 20 bytes); `parseChipRecord(pages): ByteArray?` (validates length + 3 magic bytes + high-nibble `version == CHIP_FORMAT_VERSION` (forward-incompat guard) + low-nibble `type == CHIP_TYPE_KP` (КП-only reader); returns the 16-byte code or null — any mismatch incl. unknown version/future participant type → null); `chipModelFromVersion(resp): String` (8-byte GET_VERSION → "NTAG213"/"215"/"216"/"MIFARE Ultralight"/"неизвестно", never throws); `chipCodeHex`/`chipCodeFromHex`. Constants are **`Int`** (`MAGIC`, `CHIP_TYPE_KP = 0x1`, `CHIP_TYPE_PARTICIPANT = 0x2` reserved/unused, `CHIP_FORMAT_VERSION = 0x1`, `HEADER_PAGE = 4`, `CODE_PAGE_START = 5`, `CHIP_RECORD_BYTES = 20`, `CHIP_CODE_BYTES = 16`) so nibble math and the `Int type` param don't clash with `Byte`. **Transport seam** `fun interface NfcTransport { fun transceive(frame): ByteArray }` makes the command sequencing JVM-testable with a fake (the real adapter is `{ frame -> nfcA.transceive(frame) }`). `internal fun writeRecord(t, record): ChipWriteResult` writes in **header-last** (commit-marker) order — (1) invalidate page 4 with an all-zero header, (2) write code pages 5–8, (3) write the valid header page 4 last — so an interrupted tap can never leave a valid magic over a half-written code; it then **reads back over the same open transport** (NOT a second `connect()` — only one `TagTechnology` connection is allowed at a time) and returns `Failed` unless the parsed code round-trips. `internal fun readRecord(t): ByteArray?` tries **FAST_READ** (`0x3A 04 08` → 20 bytes in one transceive), treating an `IOException` **or** a response < 20 bytes (a 1-byte NAK) as failure and falling back to two plain **READ**s (`0x30` page 4 + page 8); each READ must return ≥ `READ_BLOCK` (16) bytes — a short/NAK/`IOException` on either → null (never throws, never indexes past the buffer). I/O wrappers (blocking, off-main, never-throw): `writeChipCode(tag, code)` (builds `buildChipRecord(CHIP_TYPE_KP, code)`, opens **one** NfcA connection, wraps as `NfcTransport`, calls `writeRecord`), `readChipCode(tag): ByteArray?` (signature preserved so scan/verify callers keep working), `readChipVersion(tag): ByteArray?` (GET_VERSION `0x60` → 8 bytes or null). `NfcA` is used directly instead of `MifareUltralight.get()` because Android omits the MifareUltralight tech for many NTAG/Ultralight chips. **No NDEF** — `writeChipCodeNdef`/`chipCodeFromNdef` and all NDEF/TLV/AAR helpers were removed (no installed base of NDEF chips → no dual-format support, no NDEF-read fallback; every chip is (re)written raw at provisioning). The pure helpers + the transport-driven `writeRecord`/`readRecord` sequencing (header-last order, ACK/NAK, FAST_READ fallback on both failure modes, fallback robustness, read-back mismatch) are JVM-unit-tested by `MifareUltralightWriterTest` (fake `NfcTransport`); the thin NfcA adapter (real `connect`/`transceive`/`close`) stays hardware-bound and untested per repo convention.
- `data/crypto/LegendCrypto.kt` — pure, offline, Android-free crypto engine for the encrypted legend (mirrors the server `src/apps/mobile/crypto.py` + `legend_crypto.py`; spec in `docs/design/API.md`). `bid(code) = sha256(code).hex().take(16)`; `deriveWrapKey(code)` is hand-rolled HKDF-SHA256 over `HmacSHA256` (extract with 32 zero-byte `salt=None`, expand with `info="kp-wrap-v1"` US_ASCII, len 32); `open(key, ivB64, ctB64, aad)` is AES-256-GCM (`Cipher("AES/GCM/NoPadding")`, `GCMParameterSpec(128, iv)`, `updateAAD`, `doFinal`); `unlock(code, tag, encById, json): UnlockResult` does the `bundle_blob → content_key → enc` indirection (open bundle with `aad=bid`, then per cpId open `enc` with `aad=str(cpId)`), **never throws** (catches `GeneralSecurityException`/`SerializationException`/`IllegalArgumentException` → `Failed`). Consumes minimal value types (`UnlockTag`, `EncBlob`) — not DTOs/entities — so the repo builds the maps. **Base64 is okio `ByteString.decodeBase64()`**, not `java.util.Base64` (needs API 26 > minSdk 24) or `android.util.Base64` (stubbed in JVM tests). Result types `UnlockResult { Revealed, IdentityOnly, Failed }`, `RevealedCheckpoint`, `EncBlob`, `UnlockTag` are top-level in this file. A `LegendCryptoSanityTest` covers bid/wrap-key/round-trip/full unlock; `LegendCryptoTest` holds the server-vector assertions, `@Ignore`d until a server-generated fixture lands (the engine ships **crypto-unverified against the server** until then).

- `data/ThemePreference.kt` — reactive, persisted app theme preference (the user theme override). `ThemePreference(load: () -> String?, save: (String) -> Unit)` holds `private val _mode = MutableStateFlow(parseThemeMode(load()))` exposed as `val mode: StateFlow<ThemeMode>`; `setMode(m)` updates the flow **and** persists `m.name` via `save`. - `data/AdminAuthRepository.kt` — reactive source of truth for the race-admin session. `sealed interface AdminSession { LoggedOut | LoggedIn(email, token, expiresAt) }`. Constructor takes `apiClient` + `adminTokenStore` (+ injectable `nowUtcIso` for tests). `_session: MutableStateFlow<AdminSession>` is seeded **synchronously** from the store at construction (a stored session past its `expiresAt` is treated as `LoggedOut` and cleared), exposed as `session: StateFlow<AdminSession>`; `token(): String?` reads `_session.value` synchronously (no suspend/I/O) so the OkHttp interceptor thread can call it. `login(email, password): LoginOutcome` (`Success`/`InvalidCredentials`/`RateLimited`/`Offline`/`Error`) persists + flips the flow on success; `logout()` fires `apiClient.logout()` best-effort then **always** `store.clear()` + `LoggedOut` (so an offline logout still clears locally); `onUnauthorized()` (called on a protected-request `401`) clears + `LoggedOut`. Pure top-level helpers: `loginOutcome(result)` (status→outcome), `adminErrorMessage(outcome)` (RU strings, deliberately ambiguous «Неверный email или пароль»), and `isExpired(expiresAt, nowUtcIso)` — a **lexicographic string compare** valid because both are fixed-width UTC `yyyy-MM-dd'T'HH:mm:ss'Z'` strings (`SimpleDateFormat` in UTC, NOT date-only `todayIso()`); exact equality counts as expired. Tested by `AdminAuthRepositoryTest` (outcome mapping, logout-clears-on-`Offline`, `onUnauthorized` clears, `isExpired` boundary, seed treats past-expiry as `LoggedOut`, `adminErrorMessage` strings).
- `data/time/TrustedClock.kt` — **pure, Android-free** trusted-time core (JVM-unit-testable like `ScanSession.kt`). Holds a `ClockAnchor(serverEpochMs, anchorElapsedMs, capturedWallMs, bootCount: Int?)` and derives **trusted** time from the monotonic `elapsedRealtime()` timer: `trusted = serverEpochMs + (elapsedNow − anchorElapsedMs)` — immune to wall-clock changes (the monotonic timer isn't settings-adjustable); only a **reboot** breaks it (`elapsedRealtime` resets to 0). Top-level types: `TimeSample(wallMs, elapsedMs, trustedMs: Long?, bootCount: Int?)` (one atomic snapshot of all time sources — invariant: `elapsedMs == elapsedProvider()`, the raw `elapsedRealtime`, the **same** monotonic source the scan-window ring uses), `ClockStatus { NoSync | Ok | Skewed(skewMs) }` (`Ok` carries no payload), and `SKEW_THRESHOLD_MS = 60_000L`. Deps injected (`elapsedProvider`/`wallProvider`/`bootCountProvider`/`persist = {}`/`persisted = null`). **Warm start through boot identity (P0):** `verified = persisted != null && currentBoot != null && persisted.bootCount == currentBoot` — `null == null` does **not** grant warm start (would falsely verify a reboot); `statusFlow` is seeded in the ctor from the warm state so the UI gets the right status on the first frame (no NoSync-until-first-tick). **Threading (P1):** state is an `AtomicReference<ClockState(anchor, verified)>` for lock-free reads (`sample()`/`trusted()`/`signingSeconds()` take `.get()` once); all writes (`onServerTime`/`recomputeStatus`) run under one `synchronized(lock)` so a late network thread or a 5 s UI tick can't overwrite the store/flow with an older value. `onServerTime(serverMs, anchorElapsed, wallNow, bootNow)` reads a fresh `elapsedNow` then accepts iff: (a) no current anchor; (b) current is **monotonically invalid** (`anchorElapsedMs > elapsedNow` ⇒ reboot — stale dropped, incoming accepted unconditionally, the authoritative reboot detect that works without `BOOT_COUNT`); (c) both boot ids non-null and differ; or (d) same session and `anchorElapsed >= current.anchorElapsedMs` (a late out-of-order smaller `anchorElapsed` is rejected). On accept: ordered three-step `ref.set(new)` → `runCatching { persist(anchor) }` (best-effort — must not throw on the OkHttp thread, and must not skip the next step or `ref`/`statusFlow` desync; the only degradation on write failure is «warm start off next launch») → `statusFlow.value = …`. `computeTrusted(state, elapsedNow, bootNow)` is the pure helper (reboot-invalidate on monotonic regression OR both-boot-non-null-and-differ; a lone `null` doesn't prove reboot). `signingSeconds() = (trusted() ?: wallProvider()) / 1000` is the `ts` source for request signing (Task 4b). `trustedAt(elapsedAt: Long, bootAt: Int?): Long?` extends the trusted formula to an **arbitrary past** monotonic moment (the GPS-track use: each batched fix is anchored from its own `elapsedRealtimeNanos`, not the delivery time) — `serverEpochMs + (elapsedAt − anchorElapsedMs)`, lock-free `.get()`. Unlike `trusted()`/`computeTrusted` it does **not** treat monotonic regression as a reboot signal (a pre-anchor fix legitimately has `elapsedAt < anchorElapsedMs` — the point was captured before the network set the session's anchor; the formula extrapolates correctly backward with a negative Δ); reboot detection here is **only** boot-session identity (`bootAt != null && anchor.bootCount != null && bootAt != anchor.bootCount → null`; a lone null boot id is no reboot evidence → trust & extrapolate). `status: StateFlow<ClockStatus>` (deduped). Tested by `TrustedClockTest` (trusted formula, `signingSeconds` trusted/wall, `trustedAt` past/pre-anchor in the same boot session, `trustedAt` boot-mismatch → null, warm start by matching `bootCount`, `null==null` → NoSync, reboot → NoSync, monotonic-regression accept without `BOOT_COUNT`, initial `statusFlow` for verified persisted, out-of-order reject/accept, concurrent-`onServerTime` ordering of `persist`/`statusFlow`, `persist` throwing is swallowed, 60 s threshold boundaries 59 999/60 000/60 001, skew sign).
- `data/time/ClockAnchorStore.kt` — plain-SharedPreferences store for the `ClockAnchor` (trusted-clock warm-start seed); mirrors the `ThemePreference`/`InstallId` pattern (pure injected `load: (String) -> String?` + `save: (String, String?) -> Unit` seam + `fromSharedPreferences(context)` adapter, synchronous read). **Atomic write (P1):** the anchor is one **delimited string under a single key** `"anchor"` (`"$serverEpochMs|$anchorElapsedMs|$capturedWallMs|${bootCount ?: ""}"`, empty 4th segment = `bootCount == null`), not four keys — a process kill between `apply()`s can't leave a half-written mix. `read(): ClockAnchor?` (null if key absent or malformed: wrong field count / non-numeric segment), `write(anchor)` (one `save` → one `apply()`), `clear()`. Adapter: prefs file `"kolco24.clock"`. Tested by `ClockAnchorStoreTest` (round-trip incl. `bootCount` non-null/null, `clear` removes the key, pre-seeded read, absent/malformed → null).

**Scan overlay pattern:**
`MainActivity` wraps `Scaffold` in a `Box(Modifier.fillMaxSize())`. The `ScanScreen` is rendered after the Scaffold inside the same Box — when `showScan = true` it covers the entire screen. Dismiss via the close button or system back (handled by `BackHandler`) sets `showScan = false`. No navigation component used. The overlay is fed `roster = teamForTab?.members`, `bindings = Map<String,Int>` (`scanBindings`: `nfcUid → numberInTeam`, inverted from the binding rows but **filtered to the current roster's `numberInTeam` slots** — `rosterSlots`), `nfcAvailable`, and an `onScanTag: suspend (ScanInput, Long) -> ScanEvent`. Filtering is a correctness guard: a binding left over from a member since removed from the roster would otherwise let a stale chip substitute for an un-scanned participant and falsely complete a take (`complete` only compares `present.size` to `expectedCount`); excluding it makes the stale chip read as `UnboundChip`. The scan FAB is **gated on `teamForTab != null`** — with no resolved team (or a selection whose team row went missing) `onScanClick` routes to the team picker instead of opening an empty scan, and `onScanTag` also refuses (`BadKp`) when `scanRoster.isEmpty()`, so a missing-team selection (pointer non-null, team row gone) can't open a `expectedCount = 0` orphan take. `onScanTag` runs the impure side over a `ScanInput`: `ScanInput.Live(tag)` reads `code = readChipCode(tag)` (on `Dispatchers.IO`) + `uid = normalizeNfcUid(tag.id)`, while `ScanInput.Captured(code, uid)` (the opening tap, already read on the binder thread / launch intent) uses them directly — **no second chip read**. The shared tail (`legendRepo.unlock` when a code is present → `classifyTag(...)` → persist via `MarkRepository`) and the tap-time `now` threading are identical for both, returning the `ScanEvent` for the screen's `reduce`. A private `ScanTakeState` (plain class, `remember(showScan)` so it's fresh per overlay open) mirrors the UI session on the DB side: `markId`/`point`/`expectedCount`/`buffer`/`lastScanAt`. A `Kp` event starts a new take (new UUID) when the window expired (`now − lastScanAt > SCAN_WINDOW_MS`), no take is open, or the КП id changed — else it just re-stamps; a `Member` before any КП goes to `buffer`, after into `MarkRepository.addMember`. Persisting writes run on `container.applicationScope.async { ... }.await()` (so they outlive the closing overlay yet still hand the id back for the in-session `addMember` chain). The `marks` flow (`remember(selectedTeamId){ markRepo.observeMarks(it) ?: flowOf(emptyList()) }.collectAsState`) feeds `MarksScreen`.

**Chip-scan opens «Отметить КП» (open-on-tap + auto-close):**
Scanning a КП chip or a bound team-member bracelet **opens** the scan overlay and feeds that first tap into the session (previously the overlay opened only via the orange FAB; an idle КП tap just showed a Toast). With the raw-chip migration there is now a **single open source — the live-idle foreground path** (the NDEF cold/warm-launch sources are gone): `onTagDiscovered`'s idle branch reads `(code, uid)` once and, if `code != null` (a КП chip) `|| uid in boundUidsSnapshot` (a bound bracelet), publishes `CapturedScan(code: ByteArray?, uid: String, sample: TimeSample)` (top-level in `MainActivity.kt`; `code == null` ⇒ member bracelet; `sample` is `trustedClock.sample()` snapshotted at tap time, replacing the old `capturedAt: Long`) to the Activity-level `val nfcLaunchScan = MutableStateFlow<CapturedScan?>(null)`. (The old `openOnly` flag and the `handleNfcIntent` cold/warm publish were removed — raw chips can't cold-launch the app at all.) Two `@Volatile`/flow Activity fields support the binder-thread path: `boundUidsSnapshot: Set<String>` (the selected team's bound uids, mirrored by a host `LaunchedEffect(scanBindings) { activity.boundUidsSnapshot = scanBindings.keys }` — a **coarse open-gate** that may briefly lag a team switch; the authoritative roster-filtered `scanBindings` still governs scoring, so a stale bracelet just reads `UnboundChip`), and `pendingScan: MutableStateFlow<CapturedScan?>` (the opening tap handed to `ScanScreen` to drain). The host **dispatcher** is a `LaunchedEffect(captured, teamState)` (NOT a bare `collectAsState` + inline `if` — side effects must not run during composition, and keying on `teamState` re-fires the `Loading → Present` deferral) branching on `SelectedTeamState`: another overlay up (`showScan || teamFlowStep != None || confirmTeamId != null || showSettings || showAdmin || showProvisioning || showCheckChip || bindSlot != null || unbindSlot != null`) → drop + clear (buffering into `pendingScan` only for the narrow `showScan`-true-but-hook-not-yet-armed race); `Present` → reset all overlays (same as `onScanClick`) + `pendingScan.value = scan` + `showScan = true` + clear; `None`/`Missing` → open the comp picker (`pickerRaceId = selectedRaceId; teamFlowStep = CompPicker`) + clear; `Loading` → **return without clearing** so the next `teamState` transition re-runs the effect (a defensive guard for a live-idle tap landing mid team-switch — the original cold-launch race it solved is gone with the NDEF launch path). `ScanScreen` **drains** `pendingScan` via a continuous collect in its arm-`onTagForMark` `DisposableEffect` (see the `ui/scan/ScanScreen.kt` bullet) as a `ScanInput.Captured` with the captured `sample` (its monotonic `elapsedMs` = tap time, not drain time, so a slightly-later drain doesn't corrupt the 20 s window math; its `trustedMs`/`bootCount` persist into the take). **Auto-close** lives in `ScanScreen`: window expiry calls `currentOnClose()` after the finalize, and completion (`isComplete(session, roster.size)`) shows a ~1 s green «Готово!» beat then closes.

**Team-selection flow (enum overlay, same pattern):**
The flow reuses the scan-overlay approach instead of Navigation Compose. `MainActivity` holds three `rememberSaveable` fields: `teamFlowStep: TeamFlowStep { None, CompPicker, TeamPicker }` (a Kotlin enum is `Serializable`, no custom `Saver`), `pickerRaceId: Int?`, `confirmTeamId: Int?`. Overlays render after `ScanScreen` in the same `Box`: `CompPickerScreen` at `CompPicker`, `TeamPickerScreen` at `TeamPicker`, and `TeamSwitchSheet` (`ModalBottomSheet`) on top when `confirmTeamId != null`. A `BackHandler` (registered after the scan handler so it wins when enabled) steps the flow down: sheet (own dismiss) > TeamPicker > CompPicker > None. Confirming a team calls `teamRepository.selectTeam(...)` in `applicationScope` (not composition scope — the write must outlive the closing overlay) then collapses the whole flow.

**Settings overlay (same pattern):**
A fourth `rememberSaveable` flag `showSettings: Boolean` drives the `SettingsScreen` overlay. The «Команда» tab's «Настройки» row opens it (`onOpenSettings = { showSettings = true }`); it renders in the same `Box` **before** the CompPicker/TeamPicker/TeamSwitchSheet blocks so the picker draws on top when both are active. Its own `BackHandler(enabled = showSettings && teamFlowStep == None && confirmTeamId == null && !showScan)` closes it. «Сменить команду» inside Settings closes Settings then opens the comp picker (`showSettings = false; pickerRaceId = selectedRaceId; teamFlowStep = CompPicker`). `onScanClick` also resets `showSettings = false` so the scan FAB never opens behind a stale Settings overlay.

**Admin overlay (race-admin login + chip provisioning):**
Two `rememberSaveable` flags drive a stacked pair of overlays: `showAdmin: Boolean` (the `AdminScreen` login↔home) and `showProvisioning: Boolean` (the `ProvisioningScreen` pager, a sub-overlay opened from the admin home and drawn **above** `AdminScreen`). `val adminSession by container.adminAuthRepository.session.collectAsState()` is collected near the theme `collectAsState`; `session = adminSession` + `onOpenAdmin = { showSettings = false; showAdmin = true }` are passed into `SettingsScreen` (resetting `showSettings` so the two overlays never co-render). Both overlays render in the same `Box` **before** the CompPicker/TeamPicker blocks. A third flag `showCheckChip: Boolean` drives the read-only `CheckChipScreen` chip-verification sub-overlay, also opened from `AdminHome` and drawn above `AdminScreen` (verify and provisioning never co-open). `AdminScreen` gets `onClose = { showAdmin = false; showProvisioning = false; showCheckChip = false }` + `onOpenProvisioning = { showProvisioning = true }` + `onOpenCheckChip = { showProvisioning = false; showCheckChip = true }` (the `showProvisioning = false` reset mirrors the `showSettings = false` pattern — defensive, since AdminHome opens one at a time); `ProvisioningScreen(raceId = selectedRaceId, onClose = { showProvisioning = false })` and `CheckChipScreen(raceId = selectedRaceId, onClose = { showCheckChip = false })` (rendered after the `ProvisioningScreen` block, gated on `showCheckChip && !showScan`). Back handling: `BackHandler(enabled = showAdmin && !showProvisioning && !showCheckChip && !showScan && teamFlowStep == None && confirmTeamId == null)` closes admin, and two sibling `BackHandler`s registered **after** it (so they win when stacked) — one for `showProvisioning`, one for `showCheckChip` — close their respective sub-overlays. All three flags are reset to `false` in `onScanClick` and on team switch (`LaunchedEffect(selectedTeamId)`), alongside the other overlay resets, and `!showCheckChip` joins `!showAdmin && !showProvisioning` in the bind/unbind `BackHandler`/render guards. **Bind-chip overlay (NFC member-tag binding):**
`MainActivity` derives `bindings` with the existing null-guarded keyed-`remember` flow pattern: `remember(selectedTeamId) { selectedTeamId?.let { bindingRepo.observeForTeam(it) } ?: flowOf(emptyList()) }.collectAsState(...)`, mapped to `Map<Int, MemberChipBindingEntity>` keyed by `numberInTeam`, passed into `TeamScreen` with `nfcAvailable = (nfcState == Available)`. Before running `decideBind`, the callback checks pool readiness: a `hasSyncedPool: booleanArrayOf(false)` (keyed by `remember(activeRaceId)`, a mutable cell rather than `mutableStateOf` to avoid triggering recomposition) caches within the composition whether the pool is confirmed synced. **Trusted time (TrustedClock — clock-skew defense for take times):**
The take time of a КП used to be raw wall-clock (`System.currentTimeMillis()`), so reanchoring the phone's clock «moved» the recorded time. `TrustedClock` (`data/time/TrustedClock.kt`) instead derives a **trusted** time anchored to the monotonic `SystemClock.elapsedRealtime()` timer plus server time from the HTTP `Date` header — `trusted = serverEpochMs + (elapsedNow − anchorElapsedMs)`. A wall-clock change can't move it (the monotonic timer isn't settings-adjustable); only a **reboot** breaks the anchor (`elapsedRealtime` resets), detected by monotonic regression and/or `Settings.Global.BOOT_COUNT` mismatch. **Zero backend changes** — the `Date` header is free server time. Wiring: `ServerTimeInterceptor` (inner OkHttp interceptor, see its bullet) re-anchors on every network response; `ClockAnchorStore` persists the anchor (single delimited key) for a true warm start (same boot session ⇒ trusted time available immediately on a process restart, no network); `AppContainer` reads `BOOT_COUNT` once per process and owns the `TrustedClock`/store lazies. A take now records five times via `TimeSample` → `MarkRepository.startKpTake`: `trustedTakenAt` (scoring + order), `takenAt`/`updatedAt` (raw wall), `elapsedRealtimeAt` + `bootCount` (monotonic mark + boot session for future Δ forensics) — see `MarkEntity` (Room v10) and `MarkDao.observeForTeam`'s `COALESCE(trustedTakenAt, takenAt)` ordering. The **scan window** (`SCAN_WINDOW_MS`, ticker, expiry) runs on the monotonic `elapsedMs`, immune to clock changes. **Request signing** uses trusted time (`AppSignatureInterceptor.nowSeconds = { trustedClock.signingSeconds() }`): if the wall-clock is >±300 s off the sync would `403`; instead the inner `ServerTimeInterceptor` anchors even from the `403`'s `Date`, and the signature interceptor's GET/HEAD retry-once re-signs with trusted `ts` → `200` — the app **self-heals** while the skew banner keeps nagging «поправьте часы». UI: `ClockWarningBanner` (global, `Skewed` only) + `ScanClockBanner` (`Skewed` accent + soft `NoSync` in the scan overlay). A `MainActivity` `LaunchedEffect` ticks `recomputeStatus()` every 5 s so skew surfaces without a network event.

**GPS track recording (foreground service + dual batch upload + Precise/Economy profiles):**
A team records its race track locally with manual start/stop, in the background with the screen off, on one of two **profiles** (`TrackProfile`, persisted via `trackProfilePreference`): `Precise` (GPS HIGH_ACCURACY, 15 s, default — today's behavior) or `Economy` (HIGH_ACCURACY, 180 s — battery mode toggled from a Settings Switch; the 3-min interval duty-cycles the radio for a several-fold saving). `TrackRecordingService` (`TrackRecordingService.kt`, **foreground service `type=location`**, `START_NOT_STICKY` so a system kill does **not** silently resume — restart is UI-only) reads `raceId`/`teamId` from intent extras, **re-checks** `ACCESS_FINE/COARSE_LOCATION` first thing in `onStartCommand` (TOCTOU: a revoke between the launcher and start → `stopSelf()` without ever calling `startForeground` with the location type, which would `SecurityException` on target 34+), then `startForeground(FOREGROUND_SERVICE_TYPE_LOCATION)` with a low-importance «Запись трека» notification («Идёт запись трека · N точек» + a «Стоп» action `PendingIntent` → `ACTION_STOP` → `stopSelf`). It reads `activeProfile = container.trackProfilePreference.profile.value` and starts the engine via a private synchronous `startEngine(profile)` (`val e = LocationEngineFactory.create(this, profile); engine = e; e.start(onPoints, onError)` — starts the **captured local**, NPE-safe against a concurrent `engine = null`; does **not** pre-stop, callers own the stop). Each `RawFix` batch → `applicationScope.launch { trackRepository.insertAll(...) }` + `trackRecordingState = Recording(teamId, count)`; `onError` (provider lost / `SecurityException`) tears down so it never hangs as a fix-less «Recording». **Live-apply (mid-race profile toggle):** a `profileJob` is **(re)launched inside `onStartCommand`** (next to `countJob`, **not** once — `teardown()` cancels `serviceScope`, a later start recreates it, so a once-launched job would die on the new scope) on `Dispatchers.Main.immediate` (so `engine` writes serialize with the main-thread lifecycle + flush callbacks); it `collectLatest`s `trackProfilePreference.profile` **without `.drop(1)`** (a change landing between the initial `.value` read and subscription isn't dropped — instead it skips emissions `== activeProfile`), and on a real change does a **soft engine restart**: `flushThen(e) { if (engine !== e) return; e.stop(); restart to the LATEST .value }`. **`flushThen(e, after)`** runs `after` **exactly once** (`AtomicBoolean`) fired by the flush callback **or** a `mainHandler.postDelayed(complete, FLUSH_TIMEOUT_MS ≈ 4 s)` fallback (a never-completing `flushLocations()` can't leave the «Стоп» service stuck foreground); the callback **identity-guards** (`engine !== e`) so a stale callback from a rapid double-toggle can't stop a newer engine, and restarts to the freshly re-read `.value` so a toggle during the in-flight flush isn't lost — final state always converges via restart-to-latest (the `activeProfile` skip only suppresses last-completed repeats). **Stop / teardown flushes before stop:** `teardown()` = `flushThen(e) { if (engine === e) { e.stop(); engine = null }; finishTeardown() }` — the buffer is **delivered/enqueued** to `applicationScope` before stop (fixes the field-tested «Стоп»-drops-the-last-batch bug; durability rests on `applicationScope` outliving `stopSelf`, **not** a synchronous write barrier). `finishTeardown()` nulls `engine` defensively **and** `mainHandler.removeCallbacksAndMessages(null)` (drops any *other* `flushThen` instance's leaked timeout `Runnable`, e.g. an in-flight live-switch) **before** `serviceScope.cancel()`/`stopSelf()`, cancels `countJob`+`profileJob`, flips to `Idle`, and flushes `applicationScope.launch { trackRepository.uploadPending(raceId, teamId) }`. A bare `onDestroy()` (no `ACTION_STOP`) routes through a **best-effort** `flushThen` reusing `finishTeardown()`. **Guarantee:** «Стоп» + live-switch are lossless (buffer enqueued before stop); a bare `onDestroy()` or a hard kill may lose up to `maxDelay` (best-effort flush; Precise stays 60 s to bound it). `companion start(context, raceId, teamId)` / `stop(context)` wrap the intents. A disabled GPS toggle does **not** break `startForeground` (the service starts, just gets no fixes — the UI shows a banner + deep-link); only a missing permission blocks. `POST_NOTIFICATIONS` denial is non-fatal (the channel exists, recording runs, the notification is just hidden). `AndroidManifest.xml` declares `ACCESS_FINE/COARSE_LOCATION`, `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_LOCATION`, `POST_NOTIFICATIONS`, `<uses-feature android:name="android.hardware.location.gps" android:required="false" />` (keeps the app installable on GPS-less devices, like the NFC `required="false"`), and the `<service .TrackRecordingService android:foregroundServiceType="location" android:exported="false"/>`. Service/engines are untested per convention (the engine choice + mapping are JVM-tested in `LocationEngineFactoryTest`/`TrackModels`).
