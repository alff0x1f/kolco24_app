## Architecture

Single-activity Jetpack Compose app (minSdk 24, targetSdk 36). No ViewModel, no Navigation Compose — screen state lives in `MainActivity.kt` via `rememberSaveable`.

**Current structure:**
- `MainActivity.kt` — entry point; uses `HorizontalPager` (`rememberPagerState`) for swipe tab navigation; `NavigationBar` items call `pagerState.animateScrollToPage()`; `showScan: Boolean` via `rememberSaveable` drives the scan overlay; `SelectedTeamState { None, Loading, Missing, Present(team) }` private sealed interface drives the «Команда» tab: `produceState` switches from `Loading` (initial) to `None`/`Present`/`Missing` as `observeTeam` emits — `Loading` suppresses the empty-state flash before Room delivers the first value; `todayIso()` (the shared top-level helper in `data/DateUtils.kt`, `SimpleDateFormat` not `java.time.LocalDate` for minSdk-24/no-desugaring compatibility) is passed down to `CompPickerScreen`. `onRaceSelected` fires an **on-tap prefetch** before opening the team picker: `container.applicationScope.launch { teamRepo.refreshTeams(raceId) }` + `legendRepo.refreshLegend(raceId)` (applicationScope, NOT composition scope, so the warm-up outlives the closing comp picker) — the network starts during the screen transition. `TeamPickerScreen.onRefresh = teamRepo::refreshTeams` stays the authoritative UI-state driver; the rare duplicate GET with the on-tap prefetch is accepted (idempotent/ETag-guarded)
- `ui/marks/MarksScreen.kt` — Отметки tab; orange FAB triggers ScanScreen overlay via `onScanClick` callback
- `ui/legend/LegendScreen.kt` — Легенда tab; stateless: `LegendScreen(checkpoints: List<CheckpointEntity>, legendVisible, hasTeam, onChooseTeam, modifier)` renders three states — `!hasTeam` → **LegendNoTeam** (map glyph + «Выбрать команду» → `onChooseTeam`, same callback the Команда/Отметки empty states use); `hasTeam && !legendVisible` → **LegendLocked** (charcoal hero card, «ДО СТАРТА» badge, no CP count — the API returns `[]` while hidden so there is no real number); else the CP list fed from Room. Bold dark FilterChips, no trailing icon for unvisited CPs. The mono row label (`<стоимость>-<номер>`) pads `number.toString().padStart(2, '0')` so `5-01` doesn't regress to `5-1`. The `taken`-based dim+strike styling is retained though `taken` is always `false` this iteration (no marks feature yet)
- `ui/team/TeamScreen.kt` — Команда tab; state-hoisted (`selectedTeam: TeamEntity?`/`category`/`teamMissing` collected in `MainActivity`), `null`/missing → `TeamEmptyContent`, otherwise hero card (`startNumber` + name + «Категория X · N человек») and roster from `team.members`
- `ui/teampicker/` — team-selection flow (screens 04/04b/04c/04d): `CompPickerScreen` (pick a race), `TeamPickerScreen` (registered-teams list with search), `TeamSwitchSheet` (`ModalBottomSheet` confirm), `TeamEmptyContent` (empty/missing state), `TeamPickerLogic.kt` (pure, unit-tested helpers: `raceStatusPill`, `splitRaces`, `nearestRaceId`, `filterTeams`, `teamToken`, `displayTeamName`, `initials`, `peopleLine`; ISO-date strings compared lexicographically — no `java.time.LocalDate`, minSdk 24 without desugaring). `nearestRaceId(races, today): Int?` selects the soonest-starting current race (`effectiveEnd() >= today`, then `minByOrNull { it.date }`; ongoing wins; `null` when none current) — used for the startup prefetch. `TeamPickerScreen` is **non-blocking on a cold cache**: instead of a centered `CircularProgressIndicator`, it always renders the `LazyColumn` (with `CompContextCard`) and shows a slim `LinearProgressIndicator` flush under the `TopAppBar` (wrapped in a `Column`, so it does not scroll away) while `load == PickerLoad.Loading`; the "Пока никто не зарегистрирован" empty text is gated on `load != Loading` so a cold open never flashes a false "nobody registered". Existing empty/`Offline`/`HttpError`/`Forbidden`/retry + `staleCache`/`forbidden` snackbar branches take over once loading finishes with no data
- `ui/scan/ScanScreen.kt` — full-screen overlay for NFC checkpoint marking (A3 screen)
- `ui/theme/Color.kt` — full M3 light/dark token set; cool-grey light palette; `OrangeCta = #C65A2E`
- `ui/theme/Theme.kt` — `Kolco24Theme`; always uses the static brand palette (`LightColorScheme`/`DarkColorScheme`); no dynamic color wiring
- `ui/theme/Type.kt` — typography; `RobotoMono` (`FontFamily`, `res/font/roboto_mono_bold.ttf`) is used directly for all mono-number labels (КП row labels, «ДО СТАРТА» badge text, score tokens). The M3 `Typography` object still uses the platform default for prose text.

**Data layer** (races + teams + legend sync — signed `GET` with ETag, Room as single source of truth):
- `Kolco24App.kt` — `Application`; builds `AppContainer`. On `onCreate`, Launch A fires `refreshRaces()` fire-and-forget then **prefetches the nearest race**: reads `raceRepository.races.first()`, computes `nearestRaceId(races, todayIso())`, and (if non-null) warms it via concurrent child `launch { teamRepository.refreshTeams(nearest) }` + `legendRepository.refreshLegend(nearest)` (logs results; `nearest == null` on offline/empty → no-op, no crash). Launch B `collectLatest`s `teamRepository.selectedTeam` and on each non-null `raceId` refreshes **both** `legendRepository.refreshLegend(raceId)` and `teamRepository.refreshTeams(raceId)` concurrently via `coroutineScope { launch {} launch {} }` (`collectLatest` + `coroutineScope` so a team switch cancels both in-flight fetches before starting new ones; the persisted selection emits immediately on subscribe, so a cold start refreshes exactly once). All warm-up runs on `container.applicationScope`. Registered via `android:name=".Kolco24App"`. NB: the root composable in `MainActivity` is `Kolco24AppRoot()` (renamed to avoid clashing with this class).
- `AppContainer.kt` — manual DI (no Hilt); lazy `installId`, `OkHttpClient` + `AppSignatureInterceptor`, `Json`, `ApiClient`, `AppDatabase`, `RaceRepository`, `TeamRepository`, `LegendRepository`, plus `applicationScope = CoroutineScope(SupervisorJob() + Dispatchers.IO)`. Reads secrets from `BuildConfig`.
- `data/api/AppSignatureInterceptor.kt` — OkHttp interceptor; HMAC-SHA256 signs every request (canonical string from `docs/API.md`), adds the six `X-App-*`/`X-Install-Id` headers; `ts`/signature recomputed per call (survives retries). Pure `buildCanonical`/`sign` helpers live here.
- `data/api/ApiClient.kt` — generic `FetchResult<out T>` (`Success<T>(data, etag)`/`NotModified`/`Forbidden`/`Error(code)`; non-success branches are `FetchResult<Nothing>`). `fetchRaces(etag): FetchResult<List<RaceDto>>`, `fetchTeams(raceId, etag): FetchResult<TeamsResponse>`, and `fetchLegend(raceId, etag): FetchResult<LegendResponse>` (`GET /app/race/$raceId/legend/`) all delegate to one private `conditionalGet(url, etag, parse)`; GET on `Dispatchers.IO`, `If-None-Match` sent verbatim with quotes, 10s timeouts. `IOException`/`SerializationException` → `Error(null)` (never throws).
- `data/api/dto/RacesResponse.kt`, `data/api/dto/TeamsResponse.kt`, `data/api/dto/LegendResponse.kt` — `@Serializable` DTOs with snake_case `@SerialName`; `Json { ignoreUnknownKeys = true }`. `TeamDto.startNumber` (`@SerialName("start_number")`) is nullable with a default — the backend added it recently and `docs/API.md` does not document it yet. `LegendResponse(race: Int, checkpoints: List<CheckpointDto>)`; `CheckpointDto(id, number, cost, type, description)` is flat (no `@SerialName` needed). Hidden legend → `200` with `checkpoints: []`; `type` (`start|finish|test|kp`) is currently ignored (no per-type styling).
- `data/db/` — Room **v3** (`AppDatabase`, `exportSchema = true` → `schemas/.../{1,2,3}.json` committed, no `fallbackToDestructiveMigration` — a schema version bump without a migration will crash on upgrade). v1 tables: `RaceEntity` (table `races`, server `id` PK — also the domain model, no third layer; `isLegendVisible` disambiguates the locked/visible legend UI), `SyncMetaEntity` (table `sync_meta`, composite PK `(origin, resource)`, stores ETag per origin/resource). v2 adds team-selection tables: `CategoryEntity` (`categories`; `CategoryDto.order` field maps to `sortOrder` column — `order` is a reserved SQL word, `MIGRATION_1_2` uses `sortOrder` accordingly), `TeamEntity` (`teams`, `@Index` on `raceId`; `members: List<TeamMemberItem>` stored as a JSON column via `TeamMembersConverter` `@TypeConverter`), `SelectedTeamEntity` (`selected_team`, single row PK `id = 1`; `SelectedTeamDao.observe(): Flow<SelectedTeamEntity?>` emits `null` when the table is empty — this is the sentinel for "no team chosen yet"). v3 adds `CheckpointEntity` (`checkpoints`, `@PrimaryKey id` = server CP id, `@Index` on `raceId`, columns `number`, `cost`, `type`, `description`, `taken: Boolean = false`; doubles as the model). `CheckpointDao` mirrors `TeamDao`: `observeCheckpointsForRace(raceId)` (`ORDER BY number, id`) + `@Transaction replaceAllForRace`. Each `MIGRATION_*` is raw `CREATE TABLE`/`CREATE INDEX` whose SQL must match Room's generated schema JSON exactly (column names stay camelCase, e.g. indices `index_teams_raceId`, `index_checkpoints_raceId`) — `assembleDebug`/KSP does **not** verify this; a mismatch only surfaces at runtime when the upgraded DB opens. `MIGRATION_2_3` only adds the `checkpoints` table + index (existing tables untouched, so races/teams/selected-team survive). An instrumented `MigrationTest` (`app/src/androidTest/`, `MigrationTestHelper` + `androidx.room:room-testing`) guards each migration; run via `./gradlew connectedDebugAndroidTest` (needs an emulator/device). DAOs and the converter's wiring are not unit-tested (trivial); the pure converter round-trip is (`TeamMembersConverterTest`).
- `data/RaceRepository.kt` — `races: Flow<List<RaceEntity>>` from Room; `refreshRaces(): RefreshResult` (`Updated`/`NotModified`/`Offline`/`Forbidden`/`HttpError`). On `Success`: `replaceAll(races)` **then** `upsert(etag)` as two separate transactions — order is deliberate, a crash between them keeps fresh data with a stale ETag so the next refresh re-fetches and self-heals. ETag upsert is skipped when the response omits the `ETag` header (null), leaving any previous ETag in place.
- `data/TeamRepository.kt` — per-race teams/categories + the selected team. `teamsForRace(raceId)`/`categoriesForRace(raceId)`/`observeTeam(teamId)`/`selectedTeam` are Room flows; `refreshTeams(raceId): RefreshResult` mirrors `refreshRaces` (ETag stored in `sync_meta` under resource `"race/$raceId/teams"`, replaceAllForRace **then** ETag upsert, ETag skipped when null); `selectTeam(raceId, teamId)` upserts the single `selected_team` row. DTO→entity mapping lives here.
- `data/LegendRepository.kt` — per-race checkpoint legend. `checkpointsForRace(raceId): Flow<List<CheckpointEntity>>` is a Room flow; `refreshLegend(raceId): RefreshResult` mirrors `refreshTeams` (ETag stored in `sync_meta` under resource `"race/$raceId/legend"`, `replaceAllForRace` **then** ETag upsert, ETag skipped when null). The private `CheckpointDto.toEntity(raceId)` mapping is a flat 1:1 copy with `taken = false`. Separate from `TeamRepository` for single-responsibility (file-per-resource). `taken` is not in the legend API (it comes from NFC marks, not built yet) so it stays `false` and the score card reads `0 / total`.
- `data/InstallId.kt` — pure `getOrCreate(load, save)` for the install UUID (≤64 chars, API requirement) over an injected key-value store; thin SharedPreferences adapter.
- `data/DateUtils.kt` — shared top-level `todayIso(): String` (`SimpleDateFormat("yyyy-MM-dd", Locale.US).format(Date())`, no `java.time` for minSdk-24/no-desugaring compatibility); called by both `MainActivity` and `Kolco24App`.

**Config (secrets out of git):** `local.properties` keys `kolco24.apiBaseUrl` / `kolco24.appKeyId` / `kolco24.appSecret` → `BuildConfig` fields `API_BASE_URL` / `APP_KEY_ID` / `APP_SECRET` (see `app/build.gradle.kts`). Falls back to env `KOLCO24_API_BASE_URL` / `KOLCO24_APP_KEY_ID` / `KOLCO24_APP_SECRET` (so lint/test work in CI without the file); the build fails with a descriptive `error(...)` listing missing keys if neither is set. KSP + `kotlin.plugin.serialization` are applied on AGP 9's built-in Kotlin (do NOT apply `kotlin.android` on top — "plugin applied twice"); KSP version is pinned to `2.2.10-*`, and `android.disallowKotlinSourceSets=false` in `gradle.properties` lets KSP register generated sources.

**Scan overlay pattern:**
`MainActivity` wraps `Scaffold` in a `Box(Modifier.fillMaxSize())`. The `ScanScreen` is rendered after the Scaffold inside the same Box — when `showScan = true` it covers the entire screen. Dismiss via the close button or system back (handled by `BackHandler`) sets `showScan = false`. No navigation component used.

**Team-selection flow (enum overlay, same pattern):**
The flow reuses the scan-overlay approach instead of Navigation Compose. `MainActivity` holds three `rememberSaveable` fields: `teamFlowStep: TeamFlowStep { None, CompPicker, TeamPicker }` (a Kotlin enum is `Serializable`, no custom `Saver`), `pickerRaceId: Int?`, `confirmTeamId: Int?`. Overlays render after `ScanScreen` in the same `Box`: `CompPickerScreen` at `CompPicker`, `TeamPickerScreen` at `TeamPicker`, and `TeamSwitchSheet` (`ModalBottomSheet`) on top when `confirmTeamId != null`. A `BackHandler` (registered after the scan handler so it wins when enabled) steps the flow down: sheet (own dismiss) > TeamPicker > CompPicker > None. Confirming a team calls `teamRepository.selectTeam(...)` in `applicationScope` (not composition scope — the write must outlive the closing overlay) then collapses the whole flow.
