# Mobile App API — документация для разработчиков приложения

API для iOS/Android-приложения Kolco24. Смонтировано на `/app/*`. Все
эндпоинты — read-only `GET`, отдают JSON (`Content-Type: application/json`).

Пользовательской авторизации нет: приложение аутентифицирует **само себя**
HMAC-подписью каждого запроса. Без корректной подписи любой запрос получает
`403 {"detail": "Forbidden"}`.

> Внутренняя архитектура (зачем ETag, лиз/хэндофф двух серверов, схема
> версионирования) — в [README.md](README.md). Этот документ — про то, как
> приложению **вызывать** API.

## Эндпоинты

| Метод | Путь | Отдаёт |
|-------|------|--------|
| GET | `/app/races/` | список опубликованных гонок (ETag/304) |
| GET | `/app/race/<race_id>/teams/` | команды гонки + составы + категории (ETag/304) |
| GET | `/app/race/<race_id>/legend/` | легенда гонки — контрольные точки (ETag/304) |
| GET | `/app/race/<race_id>/sync/` | манифест версий — «что изменилось» одним запросом |

`race_id` — целочисленный `id` гонки из `/app/races/`. Неопубликованная или
несуществующая гонка → `404` (но только **после** успешной проверки подписи;
с плохой подписью будет `403`, не `404`).

Query-параметров ни у одного эндпоинта нет. Не добавляйте их: query string
входит в подписываемую строку, и сервер всё равно их игнорирует.

## Подпись запросов

### Заголовки

Каждый запрос несёт шесть заголовков:

| Заголовок | Пример | Что это |
|-----------|--------|---------|
| `X-App-Key-Id` | `android-v1` | id ключа этой сборки (выдаётся бэкендом вместе с секретом) |
| `X-App-Sig` | `3f1a…` (64 hex-символа) | HMAC-SHA256-подпись запроса, hex, lower-case |
| `X-App-Ts` | `1718200000` | unix-время устройства в **секундах** на момент запроса |
| `X-Install-Id` | `b3c4…` | стабильный id установки (UUID, генерируется при первом запуске, ≤64 символов) — **обязателен** |
| `X-App-Platform` | `ios` / `android` | платформа (≤16 символов, для статистики) |
| `X-App-Version` | `1.4.2` | версия приложения (≤32 символов, для статистики) |

`X-App-Platform` и `X-App-Version` на проверку не влияют, но присылайте их
всегда — по ним бэкенд видит, какие сборки живы (важно при ротации ключей).

### Канонная строка

Обе стороны подписывают одну и ту же строку из четырёх частей, разделённых
`\n` (LF, без завершающего):

```
METHOD \n full_path \n ts \n sha256_hex(body)
```

- `METHOD` — HTTP-метод в верхнем регистре: `GET`.
- `full_path` — путь **с query string, если она есть** (у текущих эндпоинтов
  её нет): `/app/race/8/teams/`. Без схемы и хоста. Завершающий `/`
  обязателен — Django-путь именно такой; подпишете без слеша, а отправите со
  слешем (или наоборот) — подпись не сойдётся.
- `ts` — то же значение, что в `X-App-Ts`, как строка.
- `sha256_hex(body)` — hex SHA-256 «сырого» тела запроса. У `GET` тело
  пустое, поэтому это константа:
  `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`.

Пример канонной строки:

```
GET
/app/race/8/teams/
1718200000
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
```

### Подпись

```
sig = hex( HMAC-SHA256( key = secret_utf8, message = canonical_utf8 ) )
```

Результат — 64 hex-символа; сервер приводит присланную подпись к lower-case
перед сравнением, но шлите сразу lower-case.

Секрет выбирается по `X-App-Key-Id`: в каждую сборку вшита своя пара
`(key_id, secret)`. Подменить `key_id` нельзя — сервер возьмёт другой секрет
и подпись не сойдётся, так что key_id неявно «привязан» подписью.

### Проверка на устройстве (Swift / Kotlin)

```swift
import CryptoKit

let ts = String(Int(Date().timeIntervalSince1970))
let bodyHash = SHA256.hash(data: Data()).map { String(format: "%02x", $0) }.joined()
let canonical = ["GET", "/app/race/8/teams/", ts, bodyHash].joined(separator: "\n")
let key = SymmetricKey(data: Data(secret.utf8))
let sig = HMAC<SHA256>.authenticationCode(for: Data(canonical.utf8), using: key)
    .map { String(format: "%02x", $0) }.joined()
```

```kotlin
val ts = (System.currentTimeMillis() / 1000).toString()
val bodyHash = MessageDigest.getInstance("SHA-256").digest(ByteArray(0))
    .joinToString("") { "%02x".format(it) }
val canonical = listOf("GET", "/app/race/8/teams/", ts, bodyHash).joinToString("\n")
val mac = Mac.getInstance("HmacSHA256").apply {
    init(SecretKeySpec(secret.toByteArray(), "HmacSHA256"))
}
val sig = mac.doFinal(canonical.toByteArray()).joinToString("") { "%02x".format(it) }
```

Проверить реализацию можно против эталона на сервере —
`src/apps/mobile/signing.py` (`build_canonical` + `sign`).

### Окно времени (replay-защита)

`X-App-Ts` должен попадать в окно **±300 секунд** от серверного времени.
Часы устройства, ушедшие больше чем на 5 минут, дадут `403`. Берите время
непосредственно перед отправкой (не кешируйте), а при ретраях пересчитывайте
`ts` и подпись заново. Nonce-а нет — в пределах окна запрос можно повторить,
для read-only GET это приемлемо.

### Ошибка `403`

Любая проблема аутентификации — отсутствующий заголовок, неизвестный
`key_id`, кривой/просроченный `ts`, несошедшаяся подпись — возвращает один и
тот же ответ без деталей:

```json
{"detail": "Forbidden"}
```

Это сознательно: сервер не подсказывает, какая именно проверка провалилась.
При отладке сверяйте по чек-листу:

1. Все шесть заголовков на месте? (`X-Install-Id` обязателен.)
2. `ts` в секундах (не миллисекундах!) и часы устройства не уехали?
3. Канонная строка собрана из **того же** пути, что реально уходит в запрос
   (со слешем в конце, с query string, если есть)?
4. Подпись — hex (не Base64), от UTF-8-байтов канонной строки, секретом
   именно этой сборки?

На стороне бэкенда каждая 403 логируется с причиной (таблица
`AppAuthFailure` в админке) — если чек-лист не помог, попросите бэкендера
посмотреть `reason` по вашему IP.

## Условный GET: ETag / If-None-Match

`races`, `teams` и `legend` поддерживают условные запросы — используйте их
для фонового опроса, чтобы не качать неизменившиеся данные.

1. Первый запрос: `200`, в ответе заголовок `ETag: "a1b2c3d4e5f6a7b8"`.
   Сохраните тело **и** значение ETag (целиком, с кавычками).
2. Следующие запросы: шлите `If-None-Match: "a1b2c3d4e5f6a7b8"`.
   - Данные не менялись → `304 Not Modified`, тело пустое, ETag тот же.
   - Данные изменились → `200` с полным свежим телом и новым ETag.

Детали:

- ETag **сильный**, сравнивается байт-в-байт. Возвращайте его эхом ровно как
  получили, с кавычками: `If-None-Match: "a1b2c3d4e5f6a7b8"`.
- `If-None-Match` в канонную строку **не входит** — подпись от него не
  зависит.
- Дельта-синка нет: при `200` всегда приходит **полный** список — замещайте
  локальную копию целиком. Удалённые записи просто отсутствуют в новом
  списке.
- ETag у двух серверов (cloud / локальный на гонке) свои — храните их
  раздельно, по origin.

## Рекомендуемый цикл фонового обновления

Вместо опроса каждого ресурса дёргайте один манифест:

```
GET /app/race/8/sync/
```

```json
{
  "race": 8,
  "data_source": "cloud",
  "lease_expires_at": null,
  "versions": {
    "teams": "a1b2c3d4e5f6a7b8",
    "legend": "0f9e8d7c6b5a4321"
  }
}
```

`versions.teams` / `versions.legend` — те самые отпечатки, которые ETag
соответствующего ресурса оборачивает в кавычки: ETag = `'"' + version + '"'`.
Версии списка гонок в манифесте **нет** (манифест пер-гоночный, а список
глобальный) — `/app/races/` опрашивайте напрямую своим `If-None-Match`.

Цикл:

1. `GET sync` — один дешёвый запрос.
2. Сравните `versions.*` с сохранёнными версиями. Совпадает — ничего не
   качаем.
3. Изменилось — полный `GET` только изменившегося ресурса (с
   `If-None-Match` на случай гонки между шагами 1 и 3).

`data_source` / `lease_expires_at` — задел под схему «локальный сервер на
гонке»: сейчас `data_source` всегда отражает тип сервера (`"cloud"`), а
`lease_expires_at` всегда `null`. Парсите оба поля, но логику лиза пока не
стройте — целевая схема описана в [README.md](README.md), о её активации
бэкенд сообщит отдельно.

## Форматы ответов

### `GET /app/races/`

```json
{
  "races": [
    {
      "id": 8,
      "name": "Кольцо24 2026",
      "slug": "kolco24-2026",
      "date": "2026-06-20",
      "date_end": "2026-06-21",
      "place": "Сосновый бор",
      "reg_status": "open",
      "is_legend_visible": true
    }
  ]
}
```

- Только опубликованные гонки, отсортированы по дате убыванию (новые первые).
- `date` / `date_end` — `YYYY-MM-DD`.
- `reg_status` — одно из: `"upcoming"` (регистрация не стартовала), `"open"`,
  `"sold_out"`.
- `is_legend_visible` — `false` означает, что `legend` этой гонки сейчас
  отдаёт пустой список (см. ниже).
- Без пагинации. ETag/304 поддерживается (см. выше); версия меняется при
  правке, публикации или снятии с публикации любой гонки — в том числе при
  автоматическом переходе `reg_status` в `"sold_out"`.

### `GET /app/race/<race_id>/legend/`

```json
{
  "race": 8,
  "checkpoints": [
    {"id": 101, "number": 1, "cost": 1, "type": "start", "description": "Старт, поляна"},
    {"id": 102, "number": 31, "cost": 3, "type": "kp", "description": "Родник у тропы"}
  ]
}
```

- `id` — внутренний id КП (стабильный, удобен как ключ при синхронизации).
- Отсортировано по `number` (затем по `id` при дублях номера).
- `type` — одно из: `"start"`, `"finish"`, `"test"`, `"kp"`. Черновики
  (`draft`) не отдаются никогда.
- `number` и `cost` — целые; `description` — строка (может быть пустой).
- Если легенда гонки скрыта (`is_legend_visible: false`) — придёт `200` с
  `"checkpoints": []` (не 403 и не 404). ETag при этом тоже приходит и
  изменится, когда легенду откроют, — продолжайте опрашивать как обычно.
- NFC-теги КП легенда **не** содержит (запланировано отдельно).

### `GET /app/race/<race_id>/teams/`

```json
{
  "race": 8,
  "categories": [
    {"id": 45, "code": "m4", "short_name": "М4", "name": "Мужчины, 4 чел.", "order": 1}
  ],
  "teams": [
    {
      "id": 123,
      "teamname": "Лесные звери",
      "category2": 45,
      "ucount": 4,
      "paid_people": 4.0,
      "start_time": 0,
      "finish_time": 0,
      "members": [
        {"name": "Иванов Иван", "number_in_team": 1},
        {"name": "Петрова Анна", "number_in_team": 2}
      ]
    }
  ]
}
```

- Отсортировано по `id` (стабильный порядок, не алфавит).
- `categories` — справочник категорий гонки (отдельного эндпоинта нет),
  отсортирован по `order`, затем `id`. Включает и снятые с публикации
  категории — `category2` любой команды гарантированно резолвится по этому
  списку. Изменения категорий двигают общий ETag ресурса.
- `category2` — целочисленный id категории из блока `categories` (или
  `null`, если не назначена; такие команды в выдачу не попадают — выборка
  идёт через категорию гонки).
- `ucount` — заявленный размер команды (целое); `paid_people` — оплачено
  человек (**число с плавающей точкой**, например `4.0`).
- `start_time` / `finish_time` — unix-время в **миллисекундах**, `0` = не
  стартовали / не финишировали.
- `members` отсортированы по `number_in_team`. У участников нет id —
  идентифицируйте слот по `number_in_team`.
- Удалённые команды в список не попадают.

### `GET /app/race/<race_id>/sync/`

См. [цикл фонового обновления](#рекомендуемый-цикл-фонового-обновления).
Без ETag/304 — манифест и так дешёвый.

## Ключи и ротация секрета

- Пара `(key_id, secret)` выдаётся бэкендом на каждую сборку/платформу
  (например `android-v1`, `ios-v1`). Секрет вшивается в бинарь — храните его
  как умеете (обфускация приветствуется), но считайте барьером от «curl с
  дивана», а не от целевого реверса.
- Ротация: бэкенд добавляет новый ключ (`*-v2`), новая сборка выходит с ним,
  старая продолжает работать на `*-v1`. Старый ключ отключают, когда по
  статистике установок его никто не использует. От приложения ничего не
  требуется, кроме честных `X-App-Key-Id` / `X-App-Version` в каждом запросе.
- Получили массовые 403 на старой сборке — вероятно, её ключ уже отозван;
  единственный выход для пользователя — обновить приложение. Закладывайте
  на этот случай понятный экран «обновите приложение».
