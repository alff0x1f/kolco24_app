# Цвет КП — задача для мобильной разработки

Бэкенд добавил поле **`color`** на каждую КП и отдаёт его в эндпоинте легенды.
Этот документ — то, что нужно знать мобильным разработчикам, чтобы поддержать
цвет в приложении.

## Зачем

Для мультигонки (пеший / вело / вода) организатор хочет визуально группировать КП
по этапу/дисциплине. Раньше все КП рисовались одинаково. Теперь у КП есть
именованный цвет, который приложение само маппит на свою палитру/тему при отрисовке
карты и легенды.

## Что именно отдаёт сервер

`color` — это **не hex и не RGB**, а именованный токен. Приложение само решает,
в какой реальный цвет его покрасить.

Возможные значения:

| токен      | смысл (label на сервере) |
|------------|--------------------------|
| `""`       | Без цвета (значение по умолчанию) |
| `"red"`    | Красный    |
| `"blue"`   | Синий      |
| `"green"`  | Зелёный    |
| `"yellow"` | Жёлтый     |
| `"orange"` | Оранжевый  |
| `"purple"` | Фиолетовый |

- Поле есть **всегда** (никогда не отсутствует в ответе).
- Значение по умолчанию — пустая строка `""`. Большинство существующих КП придут
  с `color: ""` — это нормально, рисуйте их «без цвета» (нейтрально, как раньше).
- Список может расшириться в будущем. **Обрабатывайте неизвестный токен как `""`**
  (нейтральный цвет), а не падайте.

## Где это в API

Эндпоинт легенды (без изменений по адресу и аутентификации):

```
GET /app/race/<id>/legend/
```

`color` приходит **в обеих ветках** ответа — и для открытой, и для запертой КП.
Это не секрет, поэтому он есть даже там, где `cost`/`description` спрятаны под
шифрование.

Открытая КП:

```json
{ "id": 154, "number": 2, "type": "kp", "color": "blue", "cost": 4, "description": "Дерево у дороги" }
```

Запертая КП (locked, `cost`/`description` зашифрованы в `enc`):

```json
{ "id": 153, "number": 1, "type": "kp", "color": "red", "enc": { "iv": "...", "ct": "..." } }
```

То есть `color` — публичный, его можно использовать для отрисовки маркера на карте
**ещё до** физического сканирования КП и расшифровки легенды.

## Кэширование / ETag — важно

- Легенда отдаётся с сильным `ETag` и поддерживает `If-None-Match` → `304`.
- Изменение цвета КП **двигает** `ETag` легенды и `versions.legend` в манифесте
  `/app/race/<id>/sync/` — то есть приложение узнает об изменении цвета через
  обычный sync-поллинг и перекачает легенду.
- На деплое этой фичи серверная схема легенды бампнута (`_LEGEND_SCHEMA_VERSION`
  1 → 2), поэтому **все клиенты один раз получат свежую легенду** автоматически,
  даже если ни одна строка в БД не менялась. Специально ничего инвалидировать на
  клиенте не нужно.

## Что не изменилось

- Онлайн-скоринг `/api/` (`CheckpointSerializer`) `color` **не** отдаёт — это
  осознанно, скорингу цвет не нужен. Берите цвет только из легенды `/app/.../legend/`.
- Идентичность тегов (`tags`: `bid → point`), шифрование `enc`/`iv`/`ct`, member-tags —
  без изменений.

## Что нужно сделать на клиенте

1. Прочитать `color` из ответа легенды для каждой КП.
2. Завести маппинг токен → цвет своей палитры/темы (включая `""` и неизвестные → нейтральный).
3. Применить цвет при отрисовке маркеров КП на карте и в списке легенды.
4. Согласовать конкретные цвета палитры с дизайном (см. ниже).

## Согласовать с командой

- Точное соответствие токен → цвет в палитре приложения
  (`red`/`blue`/`green`/`yellow`/`orange`/`purple`/`""`). Сервер задаёт только
  семантику («красный»), визуальный оттенок — на стороне приложения.

## Полный контракт эндпоинта легенды

`GET /app/race/<id>/legend/` отдаёт легенду published-гонки. Аутентификация —
подписанный запрос мобильного приложения (HMAC, заголовки `X-App-Key-Id` +
подпись), как у остальных `/app/*`. Сильный `ETag` + поддержка `If-None-Match`
→ `304`. КП типа `hidden` («Скрытый») в ответ не попадают.

Структура ответа:

```json
{
  "race": 42,
  "checkpoints": [
    {"id": 153, "number": 1, "type": "kp", "color": "red", "enc": {"iv": "...", "ct": "..."}},
    {"id": 154, "number": 2, "type": "kp", "color": "blue", "cost": 4, "description": "Дерево у дороги"}
  ],
  "tags": [
    {"bid": "a1b2...", "point": 153, "check_method": "offline", "iv": "...", "ct": "..."},
    {"bid": "c3d4...", "point": 154, "check_method": "offline"}
  ]
}
```

### `checkpoints[]` — КП

Сериализатор ветвится по запертости КП:

- **открытая** КП → `{id, number, type, color, cost, description}`;
- **запертая** (locked) КП → `{id, number, type, color, enc}` — `cost`/
  `description` спрятаны и приходят зашифрованными в `enc = {iv, ct}`; в открытом
  виде их сервер не отдаёт.

`color` — **не секрет**, есть в **обеих** ветках (см. таблицу значений выше).

### `tags[]` — идентичность тегов и офлайн-расшифровка

На **каждый** тег приходит `{bid, point, check_method}`:

- `bid` — `sha256(code).hexdigest()[:16]`, где `code` — то, что записано в NFC-память метки;
- `point` — `cp_id` КП, к которой привязан тег (идентичность, есть всегда, и у открытых КП);
- `iv`/`ct` — присутствуют **только у тегов запертых КП** (у открытых тегов их нет).

Офлайн-поток приложения:

1. прочитать `code` из метки → посчитать `bid = sha256(code).hexdigest()[:16]`;
2. найти запись в `tags` по `bid` → `point` даёт `cp_id` физически отсканированной КП;
3. если у записи есть `iv`/`ct` (запертая КП), `HKDF(code)` расшифровывает бандл
   → `{cp_id: content_key}` → этим ключом расшифровывается `enc` каждой запертой КП.

Сырые `code`/`nfc_uid` по сети никогда не передаются.
