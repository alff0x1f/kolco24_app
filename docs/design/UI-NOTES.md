# UI layer — per-file design notes

Per-file notes for `MainActivity.kt` and `ui/**`: the "why" behind each file, its gotchas, and test coverage. Read the relevant bullet before modifying a file here; update it when behavior changes. Cross-cutting rules live in `/CLAUDE.md` («Conventions & gotchas») and are assumed, not repeated.

## Cross-file wiring (MainActivity)

- **Overlay stack** — all overlays render after `Scaffold` in one `Box`, each gated by a `rememberSaveable` flag (`showScan`, `showSettings`, `showUpload`, `showAdmin`/`showProvisioning`/`showCheckChip`/`showCheckMemberChip`/`showJudgeScan`, `bindSlot`/`unbindSlot`, `teamFlowStep`/`confirmTeamId`). `showJudgeScan: String?` is `null | "start" | "finish"` (the admin menu's two rows set it, `JudgeScanScreen` reads it as `eventType`) rather than a `Boolean`, since the same host serves both fixed-mode pages. Sibling `BackHandler`s are registered in stacking order (later wins) and each fires only when no deeper overlay is up. Flags reset on `onScanClick` and on team switch (`LaunchedEffect(selectedTeamId)`) — `showUpload` mirrors `showSettings` and is **not** reset on team switch (only reachable via Settings). Theme/economy/admin-session are collected **once** here and threaded down as params (no second `collectAsState`). **`lastRealTeamId` guard** — the app is now portrait-locked (`android:screenOrientation="portrait"` on `MainActivity`, see Config/references), which eliminates orientation-triggered Activity recreation, but other config changes (multi-window resize, font-scale) still recreate it and transiently reset `selectedTeamId`'s `collectAsState()` to `null`. A `rememberSaveable var lastRealTeamId` tracks the last genuine non-null team id; the effect body (overlay resets + the recording-stop check) early-returns unless `selectedTeamId` is non-null **and** differs from `lastRealTeamId`, so a transient-null recreate blip no longer closes open overlays (e.g. the photo-capture camera) or drops a still-relevant team's recording.
- **Scan wiring** — the overlay is fed `scanBindings` (`nfcUid → numberInTeam`) **filtered to the current roster's slots** — a correctness guard: a stale binding for a removed member would otherwise let its chip falsely complete a take (`complete` only compares `present.size` to `expectedCount`). The FAB is gated on a resolved team (else routes to the picker); `onScanTag` refuses (`BadKp`) an empty roster. The opening tap and every later tap share one `ScanInput` path (`Captured` reuses the already-read `(code, uid)`, `Live` reads them) → `legendRepo.unlock` (if a code) → `classifyTag` → persist via `MarkRepository`. **`attachLocation` fires only on the new-take branch** (one-shot `currentLocationProvider.current()`, `null` = no-op). `ScanScreen.process()` plays `scanFeedback.play(feedbackFor(event))` after the `currentOnScanTag` result so every tap (incl. idempotent re-taps) fires exactly one of the three outcomes.
- **Open-on-tap** — there is a **single open source**: the live-idle foreground path publishes `CapturedScan(code, uid, sample)` (no NDEF cold/warm launch — raw `K24` chips can't cold-launch the app). A `LaunchedEffect(captured, teamState)` dispatches it: another overlay up → drop; `Present` → open scan; `None`/`Missing` → open the picker; `Loading` → **return without clearing** so the next state transition re-fires. `ScanScreen` drains `pendingScan` so the opening tap is byte-identical to later taps. Auto-close on window expiry and on completion (a ~3.3 s green «Готово!» hold matching the completion fanfare — see the checkpoint-take celebration below).
- **Checkpoint-take celebration** — the tap that transitions a session from incomplete to complete (which can land on a **Kp** event, when members were buffered before the КП tap) plays the ordinary success scan feedback first, then `scanFeedback.checkpointCompleteFanfare()` after a short delay; `ScanScreen`'s `SUCCESS_HOLD_MS` (3 300 ms, final tick + fanfare length) holds the green «Готово!» screen for that whole sequence before auto-closing, and only that auto-close path fires `onCompleted` (never expiry or manual close). `MainActivity` sets a one-shot `pendingCelebration` (plain `remember`, not `rememberSaveable` — a recreate mid-celebration deliberately drops it; also cleared on team switch) and navigates to «Отметки» (`switchToTab(0)`). `MarksScreen` consumes `celebration` on arrival: scrolls its `LazyColumn` to the bottom, then plays `onCoinSound()` together with a bouncy scale/fade pop-in on the new last tile, and calls `onCelebrationDone()`.
- **Local-mode switch + PTR re-route** — the Settings «Данные» switch position is **derived, not stored**: `isPinned(container.raceLease.collectAsState(), selectedRaceId ?: nearestRaceId, trustedClock.trusted() ?: wallClock)`, so a handback or expiry flips it live even while Settings is open. `onLocalModeChange` launches `syncCoordinator.enterLocalMode()`/`exitLocalMode()` on `applicationScope` and maps the returned `LocalModeOutcome` to a RU Toast. «Команда»'s pull-to-refresh now calls `syncCoordinator::refreshAll` (intentionally broadens the pull from teams-only to races+teams+legend+member-tags); «Легенда»'s pull passes `source = syncCoordinator.sourceFor(raceId)` (else a pinned pull would silently `Skipped` against the cloud guard); the races picker's pull stays Cloud-only (races isn't race-scoped); `onRaceSelected`'s prefetch routes teams+legend through `sourceFor(raceId)`.

## UI layer

- `MainActivity.kt` — entry point; `HorizontalPager` swipe-tab navigation. **Hosts all overlay state** (see above) and the **NFC reader-mode infra** (`NfcAdapter.ReaderCallback`, enabled in `onResume`/disabled in `onPause`). A set of `@Volatile` tag hooks (`onTagScanned`, `onTagForChipInfo`, `onTagForMark`, `onTagForProvision`, `onTagForVerify`, `onTagForJudgeScan`) are armed/cleared by each overlay's `DisposableEffect`; `onTagDiscovered` dispatches them by priority on the binder thread (hence `@Volatile`) — `onTagForJudgeScan` sits in the admin-hook priority band, above `onTagForMark` but yielding to `onTagForVerify`, and hops to main via `mainHandler.post` (mirroring the other admin hooks) — and when none is armed the idle path reads `(code, uid)` once and publishes a `CapturedScan` (an unknown idle tag — not a КП, not a bound bracelet — instead plays `scanFeedback.neutral()` before dropping). `READER_FLAGS` carries `FLAG_READER_NO_PLATFORM_SOUNDS` to **suppress the stock Android NFC beep app-wide** — the `ScanFeedbackPlayer` outcomes replace it everywhere. **Deliberate silent paths** (not oversights): a provisioning tap during a pager swipe / busy job, and a recognized chip with no team selected (the picker navigation is the feedback). `nfcState: NfcState { NoHardware, Disabled, Available }` recomputed each `onResume`. `SelectedTeamState { None, Loading, Missing, Present }` drives the «Команда» tab (`Loading` suppresses the empty-state flash). `onRaceSelected` fires an on-tap prefetch of teams+legend on `applicationScope` so the network starts during the screen transition. A `LaunchedEffect` ticks `trustedClock.recomputeStatus()` every 5 s. **5-minute foreground upload retry:** next to that ticker, a `LaunchedEffect(Unit)` wraps `lifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) { while (true) { … ; delay(UPLOAD_RETRY_INTERVAL_MS) } }` (`UPLOAD_RETRY_INTERVAL_MS = 300_000L`) — each tick launches `trackRepo.uploadAllPending()` and `markRepo.uploadAllPending()` on `container.applicationScope` (fire-and-forget, survives backgrounding mid-batch). `repeatOnLifecycle` cancels the block when the Activity leaves `STARTED` and restarts it on return to foreground, so attempt-before-delay means a return-to-foreground is an immediate retry (deliberate); overlap with Launch B / the track service's 10-min loop / take-complete flush is safe via the repos' `Mutex.tryLock`. **60 s judge-scan upload tick:** a **separate** `LaunchedEffect(Unit)` with its own `repeatOnLifecycle(STARTED) { while (true) { … ; delay(JUDGE_SCAN_UPLOAD_INTERVAL_MS) } }` loop launches `container.judgeScanRepository.uploadAllPending()` — kept as its own coroutine rather than folded into the 5-minute loop above, since that loop's own `while(true)` never returns (a second bare `while(true)` appended in the same coroutine would be unreachable). **`uploadRefreshing` PTR host wiring:** a `remember`ed (not `rememberSaveable` — transient) `var uploadRefreshing` near `showUpload` backs `UploadScreen`'s `refreshing`/`onRefresh` params — `onRefresh` guards on `!uploadRefreshing`, runs both `uploadAllPending()` calls in a `supervisorScope` (mirrors the Launch B precedent — one target failing doesn't cancel the other) on `applicationScope`, and clears the flag in `finally` (so a thrown repo call still stops the spinner).
- `ui/marks/MarksScreen.kt` — Отметки tab on real take data; stateless. Pure `marksToTiles(marks, costOf, colorOf)` maps **one tile per `complete=true` take** (an incomplete take is kept for the server log but never shown — so an unbound roster shows nothing), oldest-first so new takes append; time from `trustedTakenAt ?: takenAt`. **Tile design:** the КП discipline color fills the **whole** flat square tile (no border/stripe/elevation) so a same-discipline cluster reads as one "blob"; an edge-to-edge 4-col grid with grout showing through 2dp seams. Pure `internal tileFill(color, darkTheme)`/`gridGrout(darkTheme)` carry a **muted, screen-scoped** palette distinct from the bright `CpColor*` bars; both take a plain `darkTheme: Boolean` and the caller resolves the **applied** theme via `MaterialTheme.colorScheme.surface.luminance() < 0.5f`, **not** `isSystemInDarkTheme()` (or a manual theme override would mismatch). `null`/unknown color → the only theme-flipping neutral fill (so a colorless tile never glows). **«ВЗЯТО» counts only scoring КП** (`cost > 0`): `takenPointCount(marks, costOf)` filters distinct `complete` takes on `costOf(it) > 0`, denominator is the server-derived `legendScoringCount` (see `LegendRepository.scoringCountForRace`) — technical CPs (`cost = 0`, e.g. test/transfer zones) never inflate the counter. Metrics from `takenPointCount`/`totalScore`; «ДО КВ» is a placeholder. **Empty state** is a state-aware next-step guide (no team / NFC off / NFC absent / chips unbound / ready), the ready branch adding a `LocationNudge` (when `!locationGranted`) and a pre-start `TrackNudge`; it is suppressed while `loading` (no skeleton — the window is a few frames) — in MainActivity the marks/bindings flows use `initial = null` **and** `flowOf(null)` (not `flowOf(emptyList())`) for the no-team branch, since that branch emits instantly and `collectAsState` keeps its value across the key change when the team resolves, which would make a list-typed initial indistinguishable from "loaded, empty"; `marksLoading = teamState is Loading || (selectedTeamId != null && (marksOrNull == null || bindingsListOrNull == null))` — the `teamState is Loading` term covers the earlier window where the selection row is still reading from Room (`selectedTeamId` still null, so the marks term is inert, while `hasTeam = teamState !is None` is already true → the ready branch «Здесь появятся отметки» would flash); bindings are included so a bound-chips zero-marks team doesn't flash «Привяжите чипы» before the correct ready branch. The team-*switch* stale-value blip stays handled by the existing `safeMarks` filter, not by this guard. **Photo additions:** `Mark` view-model carries `photoPaths: List<String>` and derived `photoCount`. **Any** take with photos (a PHOTO take **or** an NFC take that also captured evidence) renders via `PhotoTileBody` — the first frame fills the tile **edge-to-edge** (the old 3dp discipline-color perimeter border was removed; tiles separate on grout alone, the discipline color survives in the corner chips). A bare mono **«+N» extra-photo** label (no chip; the first frame is the tile, so only the hidden remainder `photoCount - 1` shows — a single-photo tile shows nothing) sits **bottom-leading** styled like the take-time caption; `PhotoTileBody(mark, fill, showCameraChip)` gates a top-**trailing** `CameraAlt` glyph chip (backing mirrors `PhotoKpChip`) to **`kind == PHOTO`** only, so an NFC-with-photos tile is told apart from a genuine photo take. `PhotoTileBody` prefers the frame's `<uuid>.thumb.jpg` (one `remember`-cached `File.exists()` stat per tile; pre-thumb frames fall back to the full frame, which Coil still downsamples to the tile size). A lightbox (`HorizontalPager`) opens on tap (view-only in Phase 1) — full frames, never thumbs; photo FAB (`CameraAlt`, `OrangeCta`) routes through `decidePhotoTarget` on click. Upload status no longer lives here — it moved to `ui/upload/UploadScreen.kt`. **Judge-review notice:** pure `photoReviewSummary(marks, costOf): PhotoReviewSummary?` — **checkpoint-level**, mirroring the metrics' `distinctBy { checkpointId }`: counts КП scored *only* by `complete` photo takes (a repeat photo take of one КП counts once; a КП that also has a complete NFC take is excluded — the chip proves the visit, as is an NFC take that merely attached photo evidence; `null` when none) — feeds `PhotoReviewNotice`, a non-clickable **warning** card under `MetricsCard` («N КП по фото (1-02, 2-03) · M баллов / Баллы засчитают после проверки судьями» — the parenthesized list names each КП by its tile token, oldest-first like the grid, zero-cost → bare number, capped at 3 by pure `tokensLabel` with a «, …» tail; plural via the shared `pluralRu`) in the app's one alert palette (`errorContainer`/`onErrorContainer`, the `ClockWarningBanner` vocabulary — deliberately no third amber hue) with a flat `error`-color square badge (white `CameraAlt`); a zero points sum (photo takes of still-locked CPs, `cost = null → 0`) drops the «· 0 баллов» part and self-corrects on reveal. **Hidden-КП notice:** pure `hiddenTakenTokens(marks, lockedIds): List<String>` (distinct complete takes of still-locked checkpoints, oldest-first, masked «?-NN» — the `?` where the cost digit would sit) feeds `HiddenKpNotice` below the photo card — same warning anatomy, `Lock` badge («N скрытых КП (?-04) / Баллы засчитают после раскрытия КП»); `lockedCheckpointIds` is derived in MainActivity from `safeCheckpoints`. A photo take of a locked КП shows in **both** cards (judge gate + unknown points) and both resolve on reveal. **Checkpoint-take celebration:** a `rememberLazyListState` backs the `LazyColumn` (previously no scroll state); on `celebration = true` a `LaunchedEffect` awaits the LazyColumn's first real layout (`snapshotFlow { listState.layoutInfo.totalItemsCount }.first { it > 0 }` — reading `totalItemsCount` immediately would race a fresh composition, e.g. this page wasn't pre-composed by the pager's `beyondViewportPageCount = 1` window), scrolls to the last item + overshoots past any tall trailing item, then fires `onCoinSound()` with a bouncy scale/fade pop-in on the new last tile via `TileGrid`'s `celebrateLastTile`/`celebrationScale`/`celebrationAlpha` params — targeted by **flat positional index** (`rowIndex * 4 + colIndex == marks.lastIndex`), **never** by `Mark` equality, since `Mark` is a value `data class` with no id field and two identical takes are `==`-equal. The whole effect body runs in a `try`/`finally` so `onCelebrationDone()` always fires, even if the effect is cancelled mid-animation by a tab switch — otherwise the host's one-shot `pendingCelebration` flag would stay stuck true and replay the celebration on the next visit. Tested by `MarksMappingTest`/`TileFillTest`.
- `ui/photo/PhotoCaptureScreen.kt` — full-screen CameraX camera overlay. Shutter button, torch toggle, horizontal thumbnail strip, per-frame delete before commit. `markId` is minted by the entry point **before** the overlay opens (chicken-and-egg: frames write under `marks/<markId>/` before the DB row exists). On commit: `AttachTo` → `attachPhotos`; `AskNumber` → `createPhotoMark`. Back with 0 frames is no-op; back with frames shows a confirm dialog then deletes written files. «изменить» (AttachTo path only) deletes current session frames before dropping into the picker. Android adapter, untested by convention. **Rotation:** since the app is portrait-locked, `previewView.display?.rotation` is permanently `ROTATION_0` and can no longer detect a landscape-held shot — `capture()` instead reads a physical-tilt tracker. Top-level pure `bucketOrientationDegrees(orientationDegrees: Int, previous: Int): Int` buckets a raw sensor reading into one of `{0, 90, 180, 270}` (90° wide bands, boundaries at the 45° midpoints), returning `previous` unchanged on `-1` (`ORIENTATION_UNKNOWN`, e.g. device flat) — JVM-tested by `PhotoCaptureScreenTest`. Private `RotationTracker(context, initialDegrees) : android.view.OrientationEventListener(context)` (seeded `0`, not a `display.rotation` read — the `PreviewView` isn't attached at `remember` time, and `display.rotation` uses the inverse sign convention from the listener's raw degrees) is `remember`ed and enabled/disabled in a `DisposableEffect(Unit)` guarded by `canDetectOrientation()`, mirroring the file's existing camera-provider bind/unbind effect; callbacks land on the constructing (main) thread, so no `@Volatile`. `capture()` converts `rotationTracker.degrees` to the matching `Surface.ROTATION_*` constant for `imageCapture.targetRotation`, replacing the old `previewView.display?.rotation` read. Untested by convention (Android adapter) apart from `bucketOrientationDegrees`.
- `ui/photo/PhotoNumberPicker.kt` — lightweight overlay with a numeric field + filtered legend list (`filterCheckpointsByQuery`). Invalid number → inline error, no orphan marks. Emits the selected `CheckpointEntity` to the host; host mints `markId` and transitions to the camera.
- `ui/legend/LegendScreen.kt` — Легенда tab; stateless. `!hasTeam` → `LegendNoTeam` CTA; else the CP list. No «ДО СТАРТА» lock card — the legend is always returned and **locked checkpoints are masked per-row** (`cp.locked` → Lock glyph, mono `?-NN`, «Откроется на КП»). The `taken` dim+strike is **team-scoped** via `takenIds: Set<Int>` from the team's complete marks (`takenPoints(marks)`) — never stored on the race-shared `CheckpointEntity`. A 4dp leading color-bar gutter (`barColor()` lives here, `null` → transparent) is purely decorative and shows pre-reveal on locked rows (color is public). **`ScoreCard`'s «$taken/$total КП» counts only scoring CPs (B2):** `internal fun CheckpointEntity.isScoring() = locked || (cost ?: 0) > 0` (a locked CP's real cost is unknown client-side, so it's assumed scoring until revealed); `LegendList` computes `takenScoring = checkpoints.count { it.id in takenIds && it.isScoring() }` and passes the server-derived `scoringCount` (param, default `0`) through as `totalScoring` — separate from `totalCount`/`takenCount` (`checkpoints.size` / all taken), which still feed the «Все N»/«Не взятые M» chips and the list itself so technical CPs (`cost = 0`) stay visible everywhere except the ScoreCard fraction. **Known accepted edge case:** a taken locked CP marked via photo before reveal has `cost = null` → its live `costOf` snapshot is `0`, so it doesn't count toward the Отметки «ВЗЯТО» counter until reveal, while `isScoring()` here already counts it (locked ⇒ scoring) — both sides self-correct on reveal.
- `ui/team/TeamScreen.kt` — Команда tab; state-hoisted. Hero card + roster from `team.members`. NFC binding is live: `bindings: Map<Int, MemberChipBindingEntity>` keyed by `numberInTeam`. `boundCount` counts from the **current roster** (not raw `bindings.size`) so a server-removed member's stale binding can't inflate it. Each `MemberRow` leads with a `StatusAvatar(bound)` (the single bound/unbound indicator); **long-press** a bound row → `onUnbindMember` (host turns it into a confirm dialog), unbound rows show a «Привязать» button (disabled when `!nfcAvailable`). «Сменить команду» moved into Settings; the «Прочее» section holds «Настройки» (`onOpenSettings`) and «Загрузка данных» (`onOpenUpload`, `CloudUpload` icon) rows.
- `ui/team/BindChipSheet.kt` — `ModalBottomSheet` driving capture → validate-against-pool → store for one member. Pure `decideBind(...)` (`NotInPool`/`AlreadyBound`/`ReadyToBind`/`AlreadyOnThisSlot`) tested by `BindChipDecisionTest`; a separate `BindSheetState` drives rendering (`Waiting`/`NotInPool`/`AlreadyBound`/`PoolNotReady`/`Success`). `Success` shows only «№{participantNumber}»; the uid surfaces only as a diagnostic on error states. The host checks pool readiness (`hasBeenSynced`) and refreshes inline (→ `PoolNotReady` spinner) before deciding. **Scan feedback fires in the host** (MainActivity), not in `BindChipSheet` itself: `NotInPool` → `failure()`; `AlreadyBound` → `neutral()` (reassign can still complete the flow); `ReadyToBind`/`AlreadyOnThisSlot` success → `success()`, DB exception → `failure()`; `onReassign` success → `success()`, exception → `failure()`; pool-refresh failure (offline/error) → `neutral()` before early return.
- `ui/settings/SettingsScreen.kt` — Settings overlay; stateless. Cards: «Команда» (`ChangeTeamRow`), «Внешний вид» (`ThemeDialog`, 3 modes), «Запись трека» (`EconomyModeRow` battery Switch — host maps `Boolean ↔ TrackProfile`), «Данные» (`LocalModeRow` — the local-mode-switch, same Switch-row idiom as `EconomyModeRow`; subtitle OFF «Обновление из интернета», busy «Обновление…» with a `CircularProgressIndicator` replacing the Switch, ON «Локальный режим до HH:mm» from `expiresAtMs`; params `localMode`/`localModeBusy`/`localModeExpiresAtMs`/`onLocalModeChange` hoisted), «Администратор» (`AdminRow`, **present in release** — subtitle «Войти»/email), and a DEBUG-only «Отладка» card (`onResetTeam`/`onClearDatabase`/`onReadChipInfo`, non-null only in debug builds).
- `ui/teampicker/` — team-selection flow (`CompPickerScreen`, `TeamPickerScreen` with search, `TeamSwitchSheet` confirm, `TeamEmptyContent`, pure `TeamPickerLogic.kt`). `TeamPickerScreen` is **non-blocking on a cold cache**: always renders the list with a slim top `LinearProgressIndicator` while loading, and gates the "nobody registered" empty text on `load != Loading` so a cold open never flashes a false empty.
- `ui/scan/ScanScreen.kt` — live stateful NFC checkpoint-marking overlay (A3). `onScanTag: suspend (ScanInput, TimeSample) -> ScanEvent`; the take persists trusted time and the 20 s window runs on the monotonic `elapsedMs` (immune to a wall-clock change). A `Mutex` serializes overlapping binder-thread taps; a `DisposableEffect` arms `onTagForMark` and on arm **drains** `pendingScan` once (the opening tap) so it is identical to later taps. A 250 ms ticker computes `remaining`; expiry finalizes then auto-closes (`currentOnClose` after releasing the mutex). `process()` plays feedback after `reduce()`: the incomplete → complete transition plays `feedbackFor(event)` for the final scan, waits `COMPLETE_FANFARE_DELAY_MS = 275L`, then plays `scanFeedback.checkpointCompleteFanfare()`; a transition guard (`!wasComplete && isComplete(...)`) means an idempotent re-tap during the hold never restarts it. **Completion** (`isComplete`) shows a `SUCCESS_HOLD_MS = 3_300L` green «Готово!» (final tick + fanfare) then closes, firing `onCompleted` (held via `rememberUpdatedState`, default `{}`) just before `currentOnClose` — expiry and manual close never call it. A FAB-opened overlay with a null session stays open until manual close.
- `ui/scan/ScanSession.kt` — pure scan-session model; defines `SCAN_WINDOW_MS = 20_000L`. `reduce(session, event, now)` is the only state machine: `Kp` drains the pre-KP buffer into `present` and re-stamps; a repeat `Kp` preserves `present`; a **KP switch** resets `present` (so the prior КП's members aren't credited to the new one); `Member` buffers while no КП else joins `present` (set-semantics). `classifyTag(...)` maps a non-null code → `Kp` (via the `unlock` outcome) or a bracelet uid → `Member`/`UnboundChip`. `isComplete(...)` is a **cosmetic dismiss gate only** — scoring is persisted independently. Tested by `ScanSessionTest`/`ScanTagDecisionTest`.
- `ui/scan/ScanFeedback.kt` — pure scan-feedback mapper. `enum ScanFeedbackKind { Success, Failure, Neutral }` + `feedbackFor(event: ScanEvent): ScanFeedbackKind` — **exhaustive over `ScanEvent` (no `else`)**: `Kp`/`Member` → `Success`, `UnboundChip`/`BadKp` → `Failure`. **`Neutral` is never returned by the mapper** — it is a non-overlay outcome produced only by direct `neutral()` calls at the idle/bind/check branch points ("duplicate, don't couple"), so the mapper stays a clean 2-way over `ScanEvent`. Tested by `ScanFeedbackTest`. The outcomes replace the suppressed platform NFC beep. **Sound is reserved for working chips** (`Success`/`Failure` play a clip); `Neutral` (unknown chip) is **vibration-only, no sound**. Completion of a full КП take is **not** modeled in this mapper — it's a session-transition check in `ScanScreen.process()` that plays the mapped final scan feedback and then calls `scanFeedback.checkpointCompleteFanfare()` directly (see the checkpoint-take celebration bullet above), keeping this mapper a clean 2-way over `ScanEvent`.
- `ui/common/ClockWarningBanner.kt` — clock-skew UI. `ClockWarningBanner` (global, under the TopAppBar) shows **only** on `Skewed`; `ScanClockBanner` (scan overlay) also surfaces the soft `NoSync` notice (the only place `NoSync` is shown — the take is written either way). Pure `formatSkewMinutes` tested by `ClockWarningBannerTest`.
- `ui/admin/AdminScreen.kt` — race-admin overlay; branches on `AdminSession`. `LoggedOut` → `LoginForm` (maps failures via `adminErrorMessage`); `LoggedIn` → `AdminHome` with «Привязать чип к КП» → provisioning, «Проверить чип КП» → check-chip, «Проверить чип участника» → member-check, «Отметка старта» / «Отметка финиша» → `onOpenJudgeScan("start"/"finish")`, «Выйти». Login/logout on `applicationScope`.
- `ui/admin/ProvisioningModel.kt` — pure model of bulk chip-provisioning. `ProvisionState`, `railTicks(...)` coverage rail, `provisionErrorMessage(result)` RU strings (note `403` covers **both** non-admin AND signature/clock failure). Tested by `ProvisioningModelTest`.
- `ui/admin/ProvisioningScreen.kt` — stateful host of the provisioning pager (one КП per page) + leaf composables. **No client-side type filter** — the server is authoritative; a non-bindable КП surfaces inline as `404`/`400`. A tap binds **only when `!pagerState.isScrollInProgress`**: `apiClient.bindTag(...)` → `writeChipCode` (raw `K24`) → `Success`; `401` → `onUnauthorized()` + close. `barColor()` is `internal` here (reused by `CheckChipScreen` in the same package). **Scan feedback fires inside the bind job** (not from `LaunchedEffect(provisionState)` — `provisioningState` is app-scoped and retains its terminal value, so a `LaunchedEffect` would replay on rotation): `Success` → `success()`; any `Failed` branch → `failure()`.
- `ui/admin/ChipCheckModel.kt` — pure model of read-only chip verification. `classifyChipCheck(...)` branches `bid == null` → `NoCode` → `tag == null` → `UnknownChip` → `checkpoint == null` → `Inconsistent` → else `Ok`. **Identity-only, fully offline** — matches the chip's derived `bid` against synced legend tags; no decrypt, no server. Tested by `ChipCheckModelTest`.
- `ui/admin/CheckChipScreen.kt` — stateful host of the verification overlay. Per tap (serialized by a `Mutex`, launched on `rememberCoroutineScope` **not** `applicationScope` — verify has no write to survive rotation): read uid+code+bid, resolve tag/cp, classify, prepend to a `recent` list capped at 20 (`removeAt(lastIndex)`, **not** `removeLast()` — avoids the minSdk-24 `NewApi` lint). Plays `container.scanFeedback` directly: `!dataReady` early-return guard → `neutral()`; `Ok` → `success()`; all other results (`UnknownChip`/`Inconsistent`/`NoCode`) → `failure()`. `UidDiff` is `internal` (reused by `CheckMemberChipScreen`, the same-package `barColor()` precedent).
