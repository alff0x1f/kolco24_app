# Выгрузка данных команды — взятия КП и GPS-трек (контракт)

> ⚠️ **Статус: клиент реализован для всех трёх эндпоинтов (`/track/`,
> `/marks/`, `/mark/<id>/photo/<frame_id>`), ждёт backend.** На бэкенде их пока
> нет. До их подъёма приложение пишет всё в локальный Room с флагами
> `uploadedLocal/uploadedCloud`/`photosUploadedLocal/photosUploadedCloud = 0`
> (данные не теряются) и периодически пытается досылать — `403`/`404`/таймаут
> просто оставляют флаг `0`. Когда эндпоинты появятся, выгрузка
> self-heal'нется без изменений в клиенте.
>
> На бэкенде **все три** эндпоинта новые. Для `/track/` JSON-контракт уже был
> зафиксирован раньше (здесь он дополнен `source_install_id` / `segment_id` /
> `altitude`); `/marks/` спроектирован с нуля — на стороне приложения дуальная
> (cloud + local LAN) идемпотентная батч-выгрузка уже работает
> (`MarkRepository`/`MarkDao`/`MarkDtos`/`ApiClient.uploadMarks`), `present[]`
> собирается из снимка участников на момент взятия. `/mark/<id>/photo/<frame_id>`
> — новый бинарный эндпоинт для кадров фото-отметок (`method="photo"` в
> `/marks/`); клиентский фильтр, ранее отсекавший фото-взятия от выгрузки,
> снят — их метаданные теперь тоже едут в `/marks/` (см. ниже).

Этот документ — про **POST-выгрузку** данных команды. Чтение легенды/команд/гонок
и схема подписи — в [API.md](API.md). Подпись/заголовки/окно времени у выгрузки
**те же**, что у GET-эндпоинтов и `bindTag` (см. ниже «Общий конверт»).

| Метод | Путь | Отдаёт |
|-------|------|--------|
| POST | `/app/race/<race_id>/marks/` | батч взятий КП команды (`method="photo"` — только метаданные) |
| POST | `/app/race/<race_id>/track/` | батч GPS-точек трека команды |
| POST | `/app/race/<race_id>/mark/<mark_id>/photo/<frame_id>` | один кадр фото-отметки (сырой JPEG) |

## Зачем provenance (мотивация)

На гонке возможны ситуации:

- **Два телефона за одну команду** пишут трек/отмечаются одновременно — ради
  надёжности (дублирование). Сервер должен **различать источник**, чтобы:
  - GPS-точки двух телефонов **не слились в одну ломаную** (раздельные треки);
  - повторная отметка той же КП с двух телефонов **не задвоила счёт**, но обе
    сохранились.
- **Кто-то выбрал чужую команду** по ошибке — данные надо принять, но иметь
  возможность **разобрать пост-фактум** (перенести на правильную команду).

Решение — везде тащить **device-provenance** (`source_install_id`) и **глобально
осмысленную идентичность** (физические чипы + номера участников), причём в
**подписанном теле**, а не только в заголовке.

## Общий конверт (для обоих эндпоинтов)

- **Те же 6 заголовков подписи** (`X-App-Key-Id`, `X-App-Sig`, `X-App-Ts`,
  `X-Install-Id`, `X-App-Platform`, `X-App-Version`), та же канонная строка
  `METHOD \n full_path \n ts \n sha256_hex(body)`, то же окно **±300 с** по
  trusted-времени, тот же `403` на плохую подпись. Тело — `application/json`, и
  его SHA-256 входит в подпись (поэтому provenance в теле — **tamper-evident**).
- **Авторизации участника нет** — это аутентификация приложения подписью, как у
  GET. Admin-bearer (как у `bindTag`) здесь **не нужен** → ветки `401` нет.
- `race_id` — в URL; `team_id` и `source_install_id` — в теле.
- **`source_install_id`** — тот же UUID, что и в заголовке `X-Install-Id`, но
  продублированный в **подписанном теле**. Это ключ «с какого телефона». Сервер
  ключует хранение по нему; заголовок остаётся для статистики/консистентности.
  Кладётся **один раз на батч** (одна выгрузка = одно устройство), не на элемент.
- **Идемпотентность**: каждый элемент несёт клиентский `id` (UUID). Сервер делает
  upsert по `id`; повторная отправка принятого элемента безопасна.
- **Две независимые цели, один контракт**: **облако** (HTTPS, тот же хост, что
  GET-эндпоинты) и **локальный сервер события** (cleartext LAN, `192.168.1.5`).
  У обеих — та же подпись. Локальный сервер обязан валидировать те же заголовки
  тем же `key_id`/`secret` (или отключить проверку времени), иначе каждый
  local-upload → `403`, и `uploadedLocal` молча останется `0`. Один `id` шлётся в
  обе цели независимо — два таргета, два флага (`uploadedLocal`/`uploadedCloud`).
- **POST не авто-ретраится** на `403` (в отличие от GET/HEAD): replay ≠
  идемпотентность на уровне auth, а серверный `403` неотличим от настоящей ошибки
  подписи. Подпись и так на trusted-времени → self-heal на следующей выгрузке.

## `POST /app/race/<race_id>/marks/` — взятия КП

**Тело** (`application/json`):

```json
{
  "team_id": 1234,
  "source_install_id": "b3c4…uuid",
  "marks": [
    {
      "id": "0f9c…uuid",
      "checkpoint_id": 264,
      "method": "nfc",
      "cp_code": "9f1a…hex16bytes",
      "cp_nfc_uid": "04A2B3C4D5E680",
      "present": [
        {"nfc_uid": "04F1E2D3C4B5A6", "code": "c3d4…hex", "number": 101, "number_in_team": 1},
        {"nfc_uid": "0411223344556677", "code": null, "number": 102, "number_in_team": 2}
      ],
      "expected_count": 4,
      "complete": true,
      "trusted_ms": 1718900000123,
      "wall_ms": 1718900000000,
      "elapsed_at": 9876543,
      "boot_count": 7,
      "location": {
        "lat": 55.7501,
        "lon": 37.6109,
        "accuracy": 6.5,
        "altitude": 184.2,
        "vertical_accuracy": 3.2,
        "gps_time_ms": 1718899999000,
        "elapsed_at": 9876100
      }
    }
  ]
}
```

Поля взятия:

- `id` — клиентский UUID (тот же `MarkEntity.id`). **Ключ идемпотентности**.
- `checkpoint_id` — id КП из легенды (`checkpoints[].id`). Номер и стоимость
  сервер берёт из своей легенды (authoritative); в теле их не дублируем.
- `method` — `"nfc"` / `"photo"`. При `"photo"` тут только метаданные —
  `cp_code`/`cp_nfc_uid` пустые, `present` пустой, `complete=true`; сервер
  видит `verified=false` по пустому `cp_code` (тот же путь, что «чужая КП» в
  разделе «Краевые случаи»). **Сами байты кадров** едут отдельным бинарным
  эндпоинтом, `POST /app/race/<race_id>/mark/<mark_id>/photo/<frame_id>` —
  см. отдельный раздел ниже.
  > ⚠️ **Требование partial-accept.** Строки `method="photo"` теперь едут в
  > **том же батче**, что обычные NFC-взятия. Эндпоинт **обязан** отвечать
  > `accepted[]` за вычетом отклонённых строк и **не должен** ронять весь
  > батч `400`-м на одной фото-строке — иначе валидные NFC-взятия того же
  > батча не отметятся вместе с ней (клиент помечает только то, что реально
  > пришло в `accepted`). Это уже общее поведение эндпоинта (см. «Частичный
  > приём» ниже), но здесь оно становится **обязательным**, а не просто
  > желательным — на клиенте больше нет фильтра, отсекающего фото-строки
  > перед отправкой.
- `cp_code` (hex, 16 байт) + `cp_nfc_uid` — **доказательство, что КП физически
  отсканирована**. `code` читается только при физическом тапе по метке, а сервер
  знает истинный `code` (сам его сгенерировал). Сырой `bid` **не шлём**: он
  публичный (есть в легенде `tags[].bid`), и по нему можно было бы форжить взятие,
  не доходя до точки. Сервер проверяет `sha256(cp_code)[:16] == bid` метки этой
  `checkpoint_id`.
- `present[]` — кто отметился, по **физической идентичности чипа**. Клиент строит
  его **слиянием** скорингового состава `MarkEntity.present` (`List<Int>`,
  источник истины по составу) со снимком участников `presentDetails`
  (`List<MarkMemberSnapshot>`, снятым на момент скана браслета — `nfc_uid` +
  `number` из привязки). Итерация идёт по `present`: где снимок есть — элемент
  обогащается, где нет (legacy-строка с `presentDetails == null`, или частичный
  снимок) — выгружается **sentinel** `{nfc_uid: null, code: null, number: 0,
  number_in_team: <num>}`. Так никто из засчитанного состава не теряется, даже на
  строках, созданных до появления снимка.
  - `nfc_uid` + `code` — UID и (когда появится на браслетах) `code` чипа. Это
    источник истины: сервер сам резолвит `uid/code → участник` по своему пулу,
    **не доверяя** клиентскому `number`. Это переживает пере-привязку чипов к
    людям посреди дистанции. `code` = `null`, пока на браслетах его нет.
    `nfc_uid = null` (вместе с `number = 0`) — **sentinel** «снимка нет», не
    реальный пустой UID.
    > Задел: на чипах участников планируется добавить `code` по аналогии с КП.
  - `number` — глобальный номер участника из пула (клиентская интерпретация,
    может быть устаревшей/чужой) — для сверки. `0` — sentinel «нет снимка».
  - `number_in_team` — номер в команде, для отображения.
- `expected_count` / `complete` — что клиент считал размером ростера и засчитанным
  взятием. Сервер хранит как «мнение клиента», но при дедупе/скоринге
  **пересчитывает сам** из `present[]` против ростера.
- `trusted_ms` — доверенное время момента отметки (скоринг/порядок); **может быть
  `null`** (нет синка часов).
- `wall_ms` — сырое стенное время момента отметки. В отличие от трека (там фоллбэк
  `gps_time_ms`), у отметки GPS-времени нет, поэтому при `trusted_ms == null` это
  **единственный** фоллбэк — шлём всегда.
- `elapsed_at` — монотонный момент отметки (`elapsedRealtimeNanos/1e6`, мс);
  вместе с `boot_count` — для возможной Δelapsed-форензики. **Может быть `null`**
  (`MarkEntity.elapsedRealtimeAt` — `Long?`; NULL отличает «нет данных» от
  настоящего `0` сразу после загрузки).
- `boot_count` — boot-сессия монотонной шкалы; **может быть `null`**.
- `location` — **античит-координата места взятия**, вложенный объект; **может быть
  `null`**, когда фикса не было (`MarkEntity.locLat == null` — нет разрешения /
  GPS выключен / таймаут / отбракованный устаревший кэш). Координата пишется
  асинхронно (одноразовый свежий фикс при скане метки КП), поэтому к моменту
  выгрузки может ещё отсутствовать — это валидно. Вложенный объект (а не плоские
  поля) выбран намеренно, чтобы `gps_time_ms`/`elapsed_at` **фикса** не путались с
  одноимёнными временами самого взятия. Поля:
  - `lat` / `lon` — координата места взятия.
  - `accuracy` — горизонтальная точность фикса, метры — **ключевой
    античит-сигнал**.
  - `altitude` — высота, метры; **может быть `null`**.
  - `vertical_accuracy` — 1-сигма оценка высоты, метры
    (`Location.verticalAccuracyMeters`); **может быть `null`**.
  - `gps_time_ms` — спутниковое время фикса; **может быть `null`**.
  - `elapsed_at` — монотонный момент **фикса** (`elapsedRealtimeNanos/1e6`, мс);
    **может быть `null`**. Разница с `mark.elapsed_at` даёт **«возраст фикса»**
    (`mark.elapsed_at − location.elapsed_at`) — второй ключевой античит-сигнал
    (свежий ли фикс снят в момент взятия или подсунут старый).

**Ответ `200`**: `{ "accepted": ["0f9c…uuid", "…"] }` — список принятых
(upsert-нутых) клиентских `id`. Приложение помечает ровно их `uploadedLocal=1` /
`uploadedCloud=1` (по цели) и шлёт следующую пачку.

## `POST /app/race/<race_id>/mark/<mark_id>/photo/<frame_id>` — кадр фото-отметки

Метаданные взятия (в т.ч. `method: "photo"`) едут обычным батчем `/marks/`
(см. выше). Сами байты кадра — отдельным, **бинарным**, не-JSON эндпоинтом,
один запрос на кадр:

- `mark_id` — `MarkEntity.id`, тот же клиентский UUID, что в `marks[].id`
  батча метаданных. Кадр может доехать только после того, как сервер увидел
  строку взятия — клиент гарантирует порядок сам (**metadata-first**): драйн
  кадров для цели стартует только когда `uploadedLocal`/`uploadedCloud` этого
  взятия для этой цели уже `1`.
- `frame_id` — стем имени файла (`<uuid>` из `marks/<mark_id>/<uuid>.jpg`, без
  `.jpg`). Стабилен и уникален на кадр — служит **ключом идемпотентности**.
- **Тело** — сырые байты JPEG (не JSON), `Content-Type: image/jpeg`.
  Даунскейл ≤1600px по длинной стороне, поворот уже запечён в пиксели (EXIF
  не нужен).
- **Подпись** — тот же конверт, что у всех остальных эндпоинтов (см. «Общий
  конверт» выше): те же 6 заголовков, канонная строка
  `METHOD \n full_path \n ts \n sha256_hex(body)`, то же окно ±300с.
  `sha256_hex(body)` считается от **сырых байт JPEG** — подпись не завязана
  на формат тела, только на байты. Admin-bearer не нужен (как у `/marks/`).
- Сервер делает **upsert по `(race_id, mark_id, frame_id)`** — повторная
  отправка уже принятого кадра безопасна (идемпотентно).
- **Ответ `200`/`201`** (тело клиент не парсит) — кадр принят. Клиент
  выставляет `photosUploadedLocal`/`photosUploadedCloud` только когда **все**
  кадры этого взятия приняты этой целью (флаг per-mark, per-target).

### Коды ответов (эндпоинт кадра)

| Код | Когда | Клиент |
|-----|-------|--------|
| `200`/`201` | кадр принят (upsert) | флаг взятия переключается, когда приняты все его кадры; драйн продолжает со следующим взятием |
| `404` | эндпоинт ещё не поднят на бэкенде, либо строка взятия ещё не долетела | **self-heal**: `photosUploaded*` остаётся `0`, повтор на следующей выгрузке; **останавливает всю цель** на этом триггере (нет смысла слать ещё N кадров, пока эндпоинт лежит) |
| `403` | подпись / окно времени | не авто-ретраится (POST); self-heal на след. выгрузке; **останавливает цель** |
| `429` | rate limit | бэкофф, дослать позже; **останавливает цель** |
| `5xx` / offline | сеть/сервер | дошлёт на следующем триггере; **останавливает цель** |
| `400` / `413` | кадр не принят (неожиданно для ≤1600px JPEG) | это взятие остаётся pending (видно в статус-строке); драйн переходит к **следующему** взятию в пачке, не блокируясь на одном «ядовитом» кадре |

Транзиентные коды (`404`/`403`/`429`/`5xx`/offline) означают «эндпоинт/сеть
сейчас недоступны» — весь драйн для этой цели останавливается в этом
триггере. `400`/`413` означают «именно этот кадр не принят» — разовая проблема
одного взятия, соседние взятия в той же пачке не должны от неё пострадать.

**Осознанный trade-off:** «ядовитый» кадр (устойчивый `400`/`413`) или
навсегда пропавший локальный файл (килл в процессе съёмки, затем sweep)
оставляет `photosUploaded*` этого взятия в `0` навсегда — видно в
статус-строке как вечно «не загружено», а не как тихая потеря. Для
доказательных фото это осознанно лучше, чем молча посчитать доехавшим то, что
сервер не получил.

## `POST /app/race/<race_id>/track/` — GPS-трек

Команда записывает трек локально (foreground service) и выгружает **батчем** (не
стримом) на стопе записи и оппортунистически при запуске.

**Тело** (`application/json`):

```json
{
  "team_id": 1234,
  "source_install_id": "b3c4…uuid",
  "points": [
    {
      "id": "0f9c…uuid",
      "segment_id": "7a2f…uuid",
      "lat": 55.75,
      "lon": 37.61,
      "altitude": 184.2,
      "vertical_accuracy": 3.2,
      "accuracy": 12.4,
      "gps_time_ms": 1718900000000,
      "trusted_ms": 1718900000123,
      "elapsed_at": 9876543,
      "boot_count": 7
    }
  ]
}
```

- `id` — клиентский UUID точки. **Ключ идемпотентности**.
- `segment_id` — UUID **сессии записи**: генерируется при тапе «Начать запись»,
  им помечена каждая точка этой сессии (старт→стоп). Сервер группирует по
  `(race_id, team_id, source_install_id, segment_id)` и **не соединяет разные
  сессии линией**. Поле **на точке** (не на батче): одна оппортунистическая
  выгрузка может нести точки нескольких прошлых сессий.
- `lat` / `lon` — координата.
- `altitude` — высота над эллипсоидом WGS84, метры (`Location.altitude`); **может
  быть `null`**, если у фикса нет вертикальной компоненты — сервер хранит как есть.
- `vertical_accuracy` — 1-сигма оценка высоты, метры
  (`Location.verticalAccuracyMeters`, API 26+); **может быть `null`** (нет оценки
  или Android < 26).
- `accuracy` — погрешность фикса, метры.
- `gps_time_ms` — спутниковый хинт времени (`Location.time`).
- `trusted_ms` — trusted-время **момента фикса** (из `elapsed_at` через
  `TrustedClock.trustedAt`); **может быть `null`** — сервер обязан принимать
  `null` (фоллбэк на `gps_time_ms`).
- `elapsed_at` — монотонный момент фикса (`elapsedRealtimeNanos/1e6`, мс); вместе
  с `boot_count` — для возможной Δelapsed-реконсиляции.
- `boot_count` — boot-сессия монотонной шкалы; **может быть `null`**.
- Локальный `wallMs` (back-projected wall-fallback) **не выгружается** — это
  локальный фоллбэк, сервер скорит по `trusted_ms`/`gps_time_ms`.

**Ответ `200`**: `{ "accepted": ["0f9c…uuid", "…"] }` — приложение помечает ровно
принятые `id` по цели и продолжает слать следующую пачку **до 500 точек**.

## Серверная модель данных (как provenance окупается)

Точное устройство таблиц — на усмотрение бэкенда; контракт подразумевает форму:

**Трек** — одна строка на точку, натуральный ключ `id` (идемпотентный upsert).
Группировка по `(race_id, team_id, source_install_id, segment_id)`. Рендер обходит
каждую группу `(команда, устройство, сессия)` по `elapsed_at` → N устройств × M
сессий = N×M ломаных, никогда не перемешанных и не соединённых через разрыв.
«Лучший трек» / «показать все» — выбор на чтении, все сырые данные сохранены.

**Отметки** — одна строка на репорт, натуральный ключ `id`. Хранит полный
provenance: `team_id` (как заявлено), `source_install_id`, `checkpoint_id`,
`cp_code`/`cp_nfc_uid`, массив `present[]` (uid/code/number/number_in_team),
времена, **античит-`location`** (координата+точность+«возраст фикса» места взятия)
и серверный `verified` (`sha256(cp_code)[:16] == bid` заявленной КП).

**Скоринг (дедуп на чтении)** — КП засчитывается команде **один раз**:
`DISTINCT checkpoint_id` среди `verified` и засчитанных-по-ростеру взятий команды.
Два телефона на ту же КП = две строки, один зачётный факт. Полнота
пересчитывается из `present[]` против ростера, а не из клиентского `complete`.

**Разбор «чужой команды» (пост-фактум)** — раз у каждой отметки глобально
осмысленная идентичность:

- Сервер сверяет `present[].number` (и далее `code`/`uid`) с ростером заявленной
  `team_id`. Если отсканированные участники не из этой команды — строка всё равно
  сохраняется, но видна в админке (например, по доле совпадения с ростером).
- Админ/скрипт может **переатрибутировать** трек+отметки устройства на правильную
  команду, переписав `team_id` для этого `source_install_id` (provenance связывает
  все данные устройства вместе).
- Физическая идентичность чипа (`cp_code`, member `code`/`uid`) делает
  переатрибуцию устойчивой к пере-привязке чипов посреди гонки — «кто реально
  был» резолвится по чипам, а не по возможно-устаревшей клиентской привязке.

## Коды ответов, частичный приём, краевые случаи

| Код | Когда | Клиент |
|-----|-------|--------|
| `200` | принято (даже частично) | помечает `accepted[]` как uploaded по цели |
| `400` | битый JSON / нет обязательного поля / `id` не UUID | дроп пачки, лог (не ретраить вслепую) |
| `403` | подпись / окно времени | **не** авто-ретраить (POST); self-heal на след. выгрузке |
| `404` | гонка не найдена (после успешной подписи) | стоп для этой гонки |
| `429` | rate limit | бэкофф, дослать позже |
| — | таймаут / IOException → `Offline` | флаг остаётся `0`, дошлёт оппортунистически |

**Частичный приём** — это и есть механизм частичных ошибок: ответ
`200 { "accepted": [...] }` перечисляет ровно принятые `id`. Клиент помечает
только их; непринятые остаются с `uploaded=0` и уйдут в следующей пачке. Пустой
`accepted` (или не-`200`) останавливает цикл без зацикливания — дошлёт при
следующем стопе/запуске.

**Краевые случаи (везде «принимаем и храним»):**

- `trusted_ms` / `boot_count` / `elapsed_at` / `altitude` / member `code` /
  `location` (целиком или любое его поле) = `null` → принимаем.
- `location == null` (нет фикса) → принимаем; отсутствие координаты — не отказ, а
  просто «нет физического доказательства присутствия» для этой строки.
- `checkpoint_id` неизвестен в гонке, но синтаксически валиден → принимаем,
  `verified=false` (данные для разбора не теряем). Только синтаксический мусор →
  `400`.
- `cp_code` не сходится с `bid` КП → принимаем, `verified=false` (не отклоняем).
- «Чужая команда» → принимаем (`verified` зависит от чипа, не от команды);
  роутер-мисматч лишь флагается для админки.
- Пустой батч → `200 { "accepted": [] }`.
- Повтор `id` → upsert, попадает в `accepted` (идемпотентно).

**Замечание про cleartext-LAN:** на `192.168.1.5` `cp_code` / member `code` **и
`location` (координата места взятия)** идут открытым текстом. Слушающий event-WiFi
может перехватить `code` и форжить присутствие, не доходя до точки, или прочитать
координаты команд. Для облака (HTTPS) проблемы нет. Для любительской гонки риск
принят (LAN-сервер считается доверенным/изолированным); challenge-response здесь —
оверкилл.
