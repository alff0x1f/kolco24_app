# Мобильное приложение: авторизация админа и привязка тегов к КП

Документ для разработчиков мобильного приложения. Описывает два сценария,
добавленных в коммите `027e91a` (feat: mobile admin add legend tags):

1. **Авторизация человека** — вход по email/паролю, получение bearer-токена,
   выход.
2. **Привязка NFC-чипа к контрольной точке (КП)** — онлайн-провижининг тегов
   администратором гонки.

Все эндпойнты находятся под префиксом `/app/` и отдаются тем же сервером, что и
остальные мобильные эндпойнты (`/app/races/`, `/app/race/<id>/legend/` и т. д.).

---

## 1. Два уровня доступа

Запросы к `/app/*` проходят через **слоистую** проверку. Важно понимать оба
уровня — они независимы.

| Уровень | Что доказывает | Как передаётся | Кто проверяет |
|---------|----------------|----------------|---------------|
| **Сборка (build HMAC)** | «Запрос пришёл из нашего официального приложения» | Заголовки `X-App-*` (подпись HMAC) | `SignedAppPermission` |
| **Личность (bearer)** | «Действует конкретный пользователь» | `Authorization: Bearer <token>` | `IsMobileUser` |
| **Право на гонку** | «Этот пользователь — админ данной гонки» | (выводится из токена) | `CanEditRaceLegend` |

- **Чтение** (`/app/races/`, `/app/race/<id>/legend/` и пр.) — только build HMAC.
- **`POST /app/login/`** — только build HMAC (токен здесь ещё не нужен — его как
  раз выдаёт этот эндпойнт).
- **`POST /app/logout/`** — build HMAC **+** bearer.
- **`POST /app/race/<id>/tags/`** — build HMAC **+** bearer **+** право админа на
  гонку.

> **Build HMAC обязателен для ВСЕХ запросов, включая login.** Bearer-токен — это
> дополнительный слой поверх него, а не замена.

## 2. Подпись запроса (build HMAC)

Каждый запрос к `/app/*` подписывается общим секретом конкретной сборки. Секрет
зашит в бинарник приложения; на сервере он выбирается по `X-App-Key-Id`.

### Обязательные заголовки

| Заголовок | Значение |
|-----------|----------|
| `X-App-Key-Id` | Идентификатор ключа сборки (выбирает секрет на сервере) |
| `X-App-Ts` | Текущее время, **unix-секунды** (целое число, строкой) |
| `X-App-Sig` | HMAC-SHA256, hex в нижнем регистре (см. ниже) |
| `X-Install-Id` | UUID установки (генерируется один раз при первом запуске, хранится локально) |

### Необязательные (для статистики)

| Заголовок | Значение |
|-----------|----------|
| `X-App-Platform` | `ios` / `android` (≤ 16 симв.) |
| `X-App-Version` | Версия приложения (≤ 32 симв.) |

### Каноническая строка и подпись

```
canonical = METHOD + "\n" + full_path + "\n" + ts + "\n" + sha256_hex(body)
sig       = hmac_sha256(secret, canonical)   // hex, нижний регистр
```

Где:

- `METHOD` — HTTP-метод в **верхнем регистре** (`GET`, `POST`).
- `full_path` — путь **вместе со строкой запроса** ровно в том виде, в каком он
  уходит на сервер (например, `/app/race/12/tags/`). Если есть query-параметры —
  включайте их.
- `ts` — то же значение, что в заголовке `X-App-Ts`.
- `sha256_hex(body)` — hex SHA-256 от **сырого тела запроса**. Для запросов без
  тела (GET) это `sha256` от пустой строки
  (`e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`).
- `body` — те же байты, что реально отправляются. Сформируйте JSON **один раз** и
  подписывайте и отправляйте одни и те же байты (иначе хеш не сойдётся).

> Подпись тела работает и для POST: подписывается **хеш** тела, а не само тело,
> чтобы канон оставался коротким.

### Окно времени (replay-защита)

`X-App-Ts` должен быть в пределах **±300 секунд** от серверного времени. Иначе —
`403`. Nonce нет; синхронизируйте часы устройства.

### Псевдокод клиента

```
ts   = String(Int(Date().timeIntervalSince1970))
body = jsonData            // Data; для GET — пустые байты
canonical = "\(method.uppercased())\n\(fullPath)\n\(ts)\n\(sha256Hex(body))"
sig  = hmacSHA256Hex(key: buildSecret, message: canonical)

request.setValue(buildKeyId, forHTTPHeaderField: "X-App-Key-Id")
request.setValue(ts,         forHTTPHeaderField: "X-App-Ts")
request.setValue(sig,        forHTTPHeaderField: "X-App-Sig")
request.setValue(installId,  forHTTPHeaderField: "X-Install-Id")
request.httpBody = body
```

### Что происходит при неверной подписи

Любая ошибка build-уровня (нет заголовков, неизвестный `key_id`, протухший `ts`,
неверная подпись, не сконфигурирован сервер) даёт **одинаковый нейтральный
ответ**:

```
403 {"detail": "Forbidden"}
```

Сервер **намеренно не сообщает**, какая именно проверка не прошла (защита от
перебора). При отладке проверяйте по порядку: часы устройства, `key_id`, что
подписываются ровно отправляемые байты тела, что `full_path` включает query.

## 3. Авторизация

### 3.1. `POST /app/login/`

Обмен email + пароля на bearer-токен. Защищён **только** build HMAC.

**Запрос**

```
POST /app/login/
Content-Type: application/json
+ заголовки X-App-* (HMAC)

{"email": "admin@example.com", "password": "..."}
```

**Успех — `200`**

```json
{
  "token": "k3Jx...<высокоэнтропийная строка>...",
  "expires_at": "2026-07-21T14:03:00Z"
}
```

- `token` — **opaque** bearer (не JWT). Показывается **один раз**. Сохраните его
  в защищённом хранилище (Keychain / Keystore). Восстановить нельзя — только
  залогиниться заново.
- Срок жизни токена — **30 дней** (`expires_at`).

**Ошибки**

| Код | Когда | Тело |
|-----|-------|------|
| `400` | Тело некорректно (нет поля, не JSON, не email) | ошибки валидации |
| `401` | Неверный пароль **или** неизвестный email | `{"detail": "Неверный email или пароль"}` |
| `403` | Не прошла подпись build HMAC | `{"detail": "Forbidden"}` |
| `429` | Превышен лимит (5 запросов/мин на IP) | — |

> `401` намеренно **одинаков** для неверного пароля и несуществующего email — это
> защита от перечисления учёток. Не показывайте пользователю «такого email нет».

### 3.2. Использование токена

Во все защищённые запросы добавляйте заголовок:

```
Authorization: Bearer <token>
```

Это **в дополнение** к заголовкам `X-App-*` — HMAC по-прежнему обязателен.

Если токен невалиден (нет/просрочен/отозван/владелец деактивирован), сервер
вернёт **`401`** (в отличие от нейтрального build-`403`):

```
401 {"detail": "Недействительный или просроченный токен"}
```

При `401` на защищённом эндпойнте — удалите локальный токен и отправьте
пользователя на повторный логин.

### 3.3. `POST /app/logout/`

Отзывает **предъявленный** токен. Требует build HMAC **+** bearer.

```
POST /app/logout/
Authorization: Bearer <token>
+ заголовки X-App-* (HMAC)
(тело пустое)
```

**Успех — `200`** (пустое тело).

- Отзывается **только** тот токен, который предъявлен в `Authorization`. Другие
  устройства того же пользователя (свои токены) продолжают работать — отзыв
  пер-устройство, а не пер-аккаунт.

## 4. Привязка тега к КП

### `POST /app/race/<race_id>/tags/`

Онлайн-эндпойнт провижининга: связывает физический NFC-чип с контрольной точкой и
возвращает `code`, который нужно записать в пользовательскую память чипа.

Требует **все три уровня**: build HMAC + bearer + право админа на гонку
(`race_id` в URL). Пользователь должен быть суперюзером или `RaceAdmin` с ролью
ADMIN для этой гонки.

**Запрос**

```
POST /app/race/12/tags/
Content-Type: application/json
Authorization: Bearer <token>
+ заголовки X-App-* (HMAC)

{"checkpoint_id": 345, "nfc_uid": "04A2B3C4D5E6"}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `checkpoint_id` | integer | Стабильный `id` КП из `checkpoints[].id` в легенде; не поле `number` |
| `nfc_uid` | string | UID отсканированного чипа, не более 255 символов |

- `nfc_uid` — UID отсканированного чипа. Сервер сам нормализует его
  (`strip` + верхний регистр), но лучше присылать уже нормализованным. Длина
  ≤ 255.

**Успех**

`201 Created` — создан новый тег. `200 OK` — тег уже был привязан к **этой же**
КП (идемпотентный повтор, дубликат не создаётся).

```json
{
  "bid": "a1b2c3d4e5f6a7b8",
  "checkpoint_id": 345,
  "number": 7,
  "nfc_uid": "04A2B3C4D5E6",
  "code": "9f86d081884c7d65"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `bid` | string | Публичный идентификатор тега |
| `checkpoint_id` | integer | Стабильный `id` привязанной КП |
| `number` | integer | Человекочитаемый номер КП |
| `nfc_uid` | string | Нормализованный UID чипа |
| `code` | string | Hex-код для записи в память NFC-чипа |

- `code` — hex-строка (16 байт). **Запишите её в пользовательскую память
  NFC-чипа.** Именно по этому коду приложение потом офлайн распознаёт КП и
  расшифровывает легенду закрытых точек.
- `bid` — публичный идентификатор тега, `sha256(code).hexdigest()[:16]`. По нему
  в легенде ищется, какая КП отсканирована. Считается из `code` детерминированно.

### Переименование в легенде

В ответе `GET /app/race/<id>/legend/` поле `point` в элементах массива `tags[]`
переименовано в `checkpoint_id`:

```diff
- {"bid": "a1b2c3d4e5f6a7b8", "point": 345, "check_method": "nfc"}
+ {"bid": "a1b2c3d4e5f6a7b8", "checkpoint_id": 345, "check_method": "nfc"}
```

`checkpoint_id` — это стабильный `id` КП, совпадающий с `checkpoints[].id`; это не
человекочитаемый `number`. Поле `point` больше не передаётся. Изменение формы
ответа сбрасывает ETag/версию легенды, поэтому клиент получит обновлённый ответ
вместо `304 Not Modified`.

**Ошибки**

| Код | Когда |
|-----|-------|
| `400` | Тело некорректно: нет `checkpoint_id`/`nfc_uid`, пустой или слишком длинный `nfc_uid` |
| `401` | Невалидный/просроченный/отозванный bearer-токен |
| `403` | Build HMAC не прошёл (`{"detail":"Forbidden"}`) **или** пользователь не админ этой гонки |
| `404` | Гонка не опубликована/не существует; КП с таким `checkpoint_id` нет в гонке; КП имеет тип `hidden` |
| `409` | Этот `nfc_uid` уже привязан к **другой** КП — `{"detail":"Этот тег уже привязан к другому КП"}` |
| `429` | Превышен лимит (60 запросов/мин на IP) |

### Логика идемпотентности и конфликтов

| Ситуация | Результат |
|----------|-----------|
| Чип ещё ни к чему не привязан | `201` — создан тег |
| Тот же чип, **та же** КП | `200` — тот же тег, дубликата нет |
| Тот же чип, **другая** КП | `409` — авто-перепривязки нет |
| КП не из этой гонки / тип `hidden` | `404` |

> Перепривязки чипа на другую КП через этот эндпойнт **нет** — это защита от
> случайной перезаписи уже провизионенного чипа. Если чип нужно переназначить —
> это делается в веб-админке.

`nfc_uid` глобально уникален (не только в пределах гонки). Даже при гонке двух
одновременных запросов сервер корректно различает «тот же чип/та же КП» (→ `200`)
и «тот же чип/другая КП» (→ `409`).

## 5. Поведение токенов (важно для клиента)

- **Срок жизни** — 30 дней с момента логина (`expires_at`).
- **Хранение** — только хеш токена лежит на сервере; сырой токен есть только у
  вас. Потеряли — перелогиньтесь.
- **Отзыв** — `logout` отзывает текущий токен. Также администратор может
  деактивировать пользователя на сервере — тогда все его токены немедленно
  перестают работать (не дожидаясь 30 дней).
- **Реакция на `401`** — на любом защищённом запросе означает «токен больше не
  годен»: чистите локальный токен, ведите на логин.

## 6. Лимиты запросов (throttling)

| Эндпойнт | Лимит |
|----------|-------|
| `POST /app/login/` | 5 / мин на IP |
| `POST /app/race/<id>/tags/` | 60 / мин на IP |

Превышение — `429`. Лимит считается по реальному IP клиента (его проставляет
nginx), подделать заголовками нельзя. Не отправляйте логины в цикле без задержки.

## 7. Примеры (curl)

Подпись в `curl` придётся посчитать заранее; примеры показывают форму запросов.

**Логин**

```bash
curl -X POST https://HOST/app/login/ \
  -H 'Content-Type: application/json' \
  -H 'X-App-Key-Id: v1' \
  -H 'X-App-Ts: 1750000000' \
  -H 'X-App-Sig: <hmac_sha256_hex>' \
  -H 'X-Install-Id: 7f3c...uuid' \
  -d '{"email":"admin@example.com","password":"secret"}'
```

**Привязка тега**

```bash
curl -X POST https://HOST/app/race/12/tags/ \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer <token>' \
  -H 'X-App-Key-Id: v1' \
  -H 'X-App-Ts: 1750000000' \
  -H 'X-App-Sig: <hmac_sha256_hex>' \
  -H 'X-Install-Id: 7f3c...uuid' \
  -d '{"checkpoint_id":345,"nfc_uid":"04A2B3C4D5E6"}'
```

## 8. Частые ошибки интеграции

- **`403 Forbidden` на всё** — почти всегда build HMAC: рассинхрон часов (выход
  за ±300 с), неверный `key_id`, подпись считается не от тех байтов тела, либо
  `full_path` без query-части. Сервер не подскажет причину — проверяйте сами.
- **Хеш тела не сходится** — JSON сериализуется дважды с разным порядком/
  пробелами. Сформируйте байты тела один раз и используйте их и для подписи, и
  для отправки.
- **`401` после успешного логина** — токен протух (30 дней), отозван или
  пользователя деактивировали. Перелогиньтесь.
- **`409`** — чип уже привязан к другой КП; перепривязка только через веб-админку.
```
