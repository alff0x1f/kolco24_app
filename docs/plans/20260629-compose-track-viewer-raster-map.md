# Compose Track Viewer + Raster Map Draft

Черновая заметка по обсуждению просмотра GPS-трека и будущей растровой карты соревнования.

## Идея

Сделать свой Compose viewer вместо Leaflet/WebView. Сначала он может показывать только трек без карты, но архитектурно лучше сразу думать о нем как о маленькой карте:

- общий viewport: zoom, pan, fit-to-track, fit-to-map;
- общая проекция `lat/lon -> world/screen`;
- слои: raster map, track, start/end markers, позже КП/подписи;
- один transform для всех слоев, чтобы трек и карта двигались/масштабировались вместе.

Это не тупик для будущей подложки. Если растр карты привязан к координатам, его можно добавить нижним слоем под тем же WebMercator-преобразованием.

## Почему не Leaflet

Leaflet через WebView возможен, но для Android/Compose это лишний слой: сложнее состояние, жесты, офлайн-файлы, тема, отладка и интеграция с текущими данными Room.

Для наших специальных карт соревнования кастомный viewer проще контролировать: рисуем только то, что нужно приложению, без полноценного web-map стека.

## Что уже есть в приложении

Запись трека уже реализована:

- `TrackPointEntity` хранит `lat`, `lon`, `accuracy`, `segmentId`;
- `TrackDao.observeForTeam(teamId, raceId)` отдает точки выбранной команды;
- `MainActivity` уже собирает `safeTrack`, `trackUsable`, `trackSegmentCount`;
- `TrackCard` сейчас показывает метрики, запись/стоп, GPX share и upload status.

То есть первый viewer - это в основном UI-задача. Запись трека, база и GPX уже есть.

## Формат карты

Проверенный пример:

- реальный файл: `assets/maps2019_sqlitedb.sqlitedb`, около 47 MB;
- `app/src/main/assets/maps2019_sqlitedb.sqlitedb` на момент проверки был пустым `0B`, его надо заменить реальной базой перед использованием в приложении.

Схема SQLite:

```sql
CREATE TABLE tiles (x int, y int, z int, s int, image blob, PRIMARY KEY (x,y,z,s));
CREATE TABLE info (maxzoom Int, minzoom Int);
```

Что внутри:

- 1671 тайл;
- PNG blobs;
- тайлы 256x256;
- `s = 0`;
- stored zoom `z = 2..6`;
- это не MBTiles, больше похоже на RMaps/BigPlanet SQLite.

Для этого файла zoom обратный относительно WebMercator:

```kotlin
webZoom = 17 - storedZ
storedZ = 17 - webZoom
```

Географический охват примерно:

```text
stored z=2, web z=15, lon 58.150635..58.502197, lat 53.930220..54.175297
stored z=3, web z=14, lon 58.139648..58.513184, lat 53.930220..54.175297
stored z=4, web z=13, lon 58.139648..58.535156, lat 53.930220..54.188155
stored z=5, web z=12, lon 58.095703..58.535156, lat 53.904338..54.213861
stored z=6, web z=11, lon 58.007812..58.535156, lat 53.852527..54.265224
```

Запрос тайла:

```sql
SELECT image FROM tiles WHERE z = ? AND x = ? AND y = ? AND s = 0
```

## Выбор формата

Лучший вариант из обсужденных - этот `.sqlitedb`, потому что это уже офлайн-тайлы и их удобно рисовать выборочно по viewport.

Остальные варианты:

- `OruxMapsImages.db + *.otrk2.xml` - тоже можно, но это более специфичный Orux формат.
- `KMZ` - хорошо для проверки/импорта или маленьких overlays, но хуже как runtime-хранилище большой карты.
- `JNX` - лучше избегать, если есть SQLite/Orux/KMZ.

Если в будущем можно переэкспортировать карты, самый чистый формат был бы MBTiles. Но текущий `.sqlitedb` уже достаточно удобен.

## Как это может работать

Вариант без карты:

- Canvas рисует трек по `safeTrack`;
- точки фильтруются через `filterPoints`;
- порядок через `sortedTrackPoints`;
- линии группируются по `segmentId`, чтобы stop/start не соединялись прямой линией;
- жесты: pinch zoom, drag pan;
- кнопки: fit/reset/fullscreen.

Вариант с картой:

- `.sqlitedb` копируется из assets во внутреннее хранилище;
- открывается read-only через `SQLiteDatabase`;
- по текущему viewport считаются видимые `x/y/z`;
- PNG тайлы читаются из `image`, декодируются и кешируются;
- тайлы рисуются нижним слоем;
- трек рисуется сверху тем же transform.

Нужен LRU cache bitmap-тайлов, иначе при pan/zoom будет лишнее декодирование PNG.

## Оценка

- Только трек без подложки: примерно 1 день для MVP, 1.5-2 дня для аккуратного варианта.
- Трек + SQLite raster map: 2-3 дня для базовой версии.
- Полированный viewer с fullscreen, cache, fit controls, хорошими edge cases: 3-5 дней.
- Если карта без координатной привязки и нужна ручная калибровка: еще 2-4 дня.

## Открытые вопросы

- Все будущие карты будут в такой же SQLite-схеме или могут быть Orux/MBTiles/KMZ?
- Константа `webZoom = 17 - storedZ` стабильна для всех экспортов или только для этого файла?
- Карта должна поставляться внутри APK/AAB, скачиваться после выбора гонки или подкладываться вручную под конкретное событие?
- Будет ли несколько карт по `raceId`?
- Есть ли координаты КП, чтобы позже рисовать их поверх карты?
